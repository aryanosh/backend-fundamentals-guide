Chapter 02

The Anatomy of a Request

HTTP methods, URL paths, headers, bodies, and the status codes that make sense of it all

~11 min read

What Actually Gets Sent Over the Wire?

In Chapter 1, we watched your phone and Twitter's server perform a TCP handshake and establish a connection. Now imagine that connection is an open phone line. The HTTP request is what you actually say into the phone. It's a carefully formatted message that tells the server exactly what you want, who you are, and often includes data you want to send. Every action you take on Twitter — viewing a profile, posting a tweet, liking a photo, sending a DM — is an HTTP request.

HTTP stands for HyperText Transfer Protocol, but don't let the name fool you — it's used for far more than just transferring hypertext. It's the language that every web browser, mobile app, API client, and server speaks. HTTP is a request-response protocol: the client (your browser or app) sends a request, and the server sends back a response. That's it. Every website, every API call, every time your phone syncs data — it's all just request-response cycles happening in rapid succession, often hundreds of them to load a single page.

Every HTTP request has three main parts: the request line (method + path + version), the headers (key-value metadata), and an optional body (the actual data). Let's break each one down.

HTTP Methods: The Verbs of the Web

The HTTP method (also called a verb) tells the server what kind of action you want to perform. Think of it as the intention behind your request. There are nine methods defined in the HTTP specification, but you'll encounter just five in everyday backend work:

GET — "Give me this resource." When you load your Twitter timeline, refresh a profile page, or view a tweet's replies, your browser sends a GET request. GET requests should never change anything on the server; they're read-only. This is called being idempotent — making the same GET request a hundred times should return the same result (assuming the data hasn't changed on the server). GET requests have no body; all the information is in the URL and headers.

POST — "Create something new." When you hit the "Post Tweet" button, your browser sends a POST request to Twitter's API with the tweet text in the body. Every time you POST, a new resource is created. POST is not idempotent — pressing "Post Tweet" twice creates two tweets (don't test this, your followers will know). POST requests carry their data in the body, typically as JSON or form data.

PUT — "Replace this resource entirely." If you edit your profile bio, Twitter might send a PUT request with your entire new profile object. PUT says "replace whatever exists at this URL with this new data." Unlike POST, PUT is idempotent — sending the same PUT request twice results in the same state on the server. The second request just overwrites with the same data.

PATCH — "Update part of this resource." Twitter might use PATCH instead of PUT when you only change your display name (not your entire profile). PATCH sends only the fields that changed, not the whole object. It's more efficient than PUT for partial updates.

DELETE — "Remove this resource." When you delete a tweet, your browser sends a DELETE request to that tweet's URL. DELETE is also idempotent — deleting a tweet that's already deleted returns the same result (often a 404 or 204 status).

Here's a concrete Twitter example. When your app loads your timeline, it sends: GET /api/v2/timeline HTTP/2. When you post a tweet, it sends: POST /api/v2/tweets HTTP/2 with a JSON body containing your tweet text. When you unlike a tweet, it sends: DELETE /api/v2/tweets/12345/like HTTP/2. Each method maps to a CRUD operation: Create (POST), Read (GET), Update (PUT/PATCH), Delete (DELETE).

URL Paths: The Address of the Resource

The path is the second part of the request line, and it tells the server which resource you're talking about. A full URL looks like https://twitter.com/elonmusk/status/123456789, but the path part is just /elonmusk/status/123456789. The path is like a file system path, but it doesn't necessarily map to actual files on the server — it maps to resources that the server's routing logic understands.

Twitter's API uses clean, hierarchical paths. For example: /api/v2/users/elonmusk gets Elon's profile. /api/v2/users/elonmusk/followers gets his followers list. /api/v2/tweets/123456789 gets a specific tweet. Notice the pattern: the path gets more specific as you go right. This hierarchy helps servers route requests to the right handler code and makes APIs intuitive for developers.

Paths can also include query parameters (after a ?) for filtering, sorting, or pagination. For example: GET /api/v2/tweets?count=20&since_id=100 says "give me 20 tweets posted since tweet ID 100." Query parameters are key-value pairs separated by &.

Headers: The Metadata That Makes It Work

Between the request line and the body, there's a block of headers. Headers are key-value pairs that provide metadata about the request or response. Think of them as the envelope around your message — they tell the server things like: what format the body is in, who's sending the request, what languages the client accepts, and authentication credentials.

Some important request headers: Authorization carries your login token (Twitter uses Bearer tokens or OAuth); Content-Type tells the server what format the body is in (usually application/json for APIs); User-Agent identifies the client app (like "Twitter/iPhone" or "Mozilla/5.0"); Accept tells the server what response formats you understand; Cookie carries session data stored by your browser.

And some response headers: Content-Type tells the client what format the response body is in; Set-Cookie tells the browser to store a cookie; Cache-Control tells the client how long to cache the response; X-Request-Id is a unique identifier for debugging (every request to Twitter gets one, so if something breaks, engineers can trace exactly which request failed).

The Body: Where the Data Lives

The body (or payload) is the actual data being sent. GET and DELETE requests typically don't have a body — they don't need one because they're either fetching or deleting an existing resource. POST, PUT, and PATCH requests usually have a body containing the data to create or update. When you post a tweet, the body might look like: {"text": "Hello world!", "media_ids": ["123"]}. When you update your profile, the body might be: {"name": "Elon", "bio": "Mars colonist"}.

The body is usually JSON in modern APIs, but it can also be XML, form-encoded data, plain text, or binary data (like image uploads). The Content-Type header tells the server how to interpret those bytes.

Status Codes: What the Server Says Back

After the server processes your request, it sends back an HTTP response with a status code. Status codes are three-digit numbers grouped into five categories. Every developer should know these by heart:

200 OK — Everything worked. The request succeeded and the response contains the requested data. When your timeline loads, you get a 200.

201 Created — A new resource was created (usually from a POST request). When your tweet posts successfully, you get a 201 back with the new tweet object in the body.

204 No Content — The request succeeded but there's nothing to return. DELETE requests often return 204.

301/302 Redirect — The resource moved. Your browser automatically follows the new URL in the Location header. Twitter uses this for URL shorteners.

400 Bad Request — You messed up. The server can't understand your request (maybe invalid JSON, missing required field). If you try to post a tweet without text, Twitter returns 400 with a message like "text is required."

401 Unauthorized — You need to log in. You tried to access a protected resource without valid credentials. If your session expires and you try to post, Twitter returns 401. It's saying "I don't know who you are."

403 Forbidden — You're logged in but you don't have permission. Maybe you tried to delete someone else's tweet. The server knows who you are but says "you're not allowed to do that."

404 Not Found — The resource doesn't exist. You tried to visit a Twitter profile that doesn't exist, or a deleted tweet.

429 Too Many Requests — You're rate-limited. Twitter's API allows a certain number of requests per minute. If you exceed it, you get 429 and need to wait.

500 Internal Server Error — The server broke. Something went wrong on Twitter's side. It's not your fault, but you also can't fix it. The Twitter engineering team gets paged when 500s spike.

502/503/504 — Gateway errors. The server is down, overwhelmed, or the upstream service timed out. If Twitter is under heavy load or a deployment goes wrong, you might see these.

Status codes are your first debugging tool. When something breaks in your backend, the status code tells you whether the problem is on your side (4xx) or the server's side (5xx). This simple distinction saves hours of debugging.

Now that you understand the anatomy of a request, you're ready for Chapter 3 — because knowing what a request looks like is just the beginning. The really interesting question is: what does the server actually do when it receives that request?
