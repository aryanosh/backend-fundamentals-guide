The Anatomy of a Request

HTTP methods, URL paths, headers, bodies, and the status codes that make sense of it all

~11 min read

HTTP Methods: The Verbs of the Web

The HTTP method tells the server what kind of action to perform. Five methods cover most real-world APIs:

  • GET — Read data. Idempotent, no body.
  • POST — Create. Carries JSON body. Not idempotent.
  • PUT — Replace entire resource. Idempotent.
  • PATCH — Partial update. Body has only changed fields.
  • DELETE — Remove resource. Idempotent.

Methods map directly to CRUD: Create, Read, Update, Delete.

The five essential HTTP methods — each has distinct semantics for idempotency and body usage

Request & Response Structure

Every HTTP exchange has two parts: a request that says what you want and a response that delivers the result. The request carries a method, path, headers, and optional body. The response carries a status code, headers, and an optional body.

Every request has a method, path, headers, and body; every response has a status, headers, and body

The Anatomy of a URL

The URL path tells the server which resource you are targeting. Paths map to resources, not files. Query parameters filter, sort, or paginate — ?q=hello&count=20 means give me 20 results matching hello.

URL anatomy, header examples, and the JSON body — the three structural parts of every request

HTTP Headers

Headers are key-value metadata between the request line and the body.

  • Request headers — Authorization (login token), Content-Type (body format), User-Agent (client identity), Accept (response format), Cookie (session data).
  • Response headers — Content-Type, Set-Cookie, Cache-Control, X-Request-Id (trace ID).
  • The Content-Type header tells the server how to parse the body bytes.

HTTP Request Body

The request body carries the actual data. GET and DELETE typically have no body. POST, PUT, and PATCH carry JSON, form data, or binary.

  • GET/DELETE — No body. The request is complete with just the method, path, and headers.
  • POST/PUT/PATCH — Body contains JSON, form data, or binary content.
  • The Content-Type header tells the server how to interpret the bytes.

What Status Codes Mean

After processing your request, the server sends back a status code — a three-digit number grouped into five categories. Status codes are your first debugging tool: 4xx means the client made a mistake, 5xx means the server broke.

Status codes are the server's way of saying what happened

The Full Request Lifecycle

After the TCP handshake establishes a connection, the client sends an HTTP request. The server processes it and sends back a response. Every request has three structural parts:

  • Request line — Method + path + HTTP version.
  • Headers — Key-value metadata (auth, content type, cookies).
  • Body — Optional payload with the actual data.
