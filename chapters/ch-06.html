Where Data Actually Lives

Relational databases, SQL, tables, keys, indexes, and when to go NoSQL

~13 min read

SQL vs NoSQL

Relational databases (SQL) store data in tables with rows and columns, enforcing a fixed schema and linking tables via foreign keys. NoSQL databases use flexible data models — documents, key-value pairs, wide columns, or graphs — trading strict consistency for scalability and schema flexibility.

  • SQL: normalized tables, ACID transactions, JOINs, fixed schema — best for core relational data with integrity constraints
  • NoSQL: flexible schema, high write throughput, fast key lookups — best for specialized or high-volume workloads
  • Polyglot persistence: use both — SQL for relational data, NoSQL for everything else
SQL uses normalized tables with foreign keys; NoSQL embeds related data in documents. Choose based on query patterns.

How Indexes Work

Without indexes, every query requires a full table scan — reading every row, taking hours on a billion-row table. An index (usually a B-tree) maps values to row locations, reducing lookup from O(n) to O(log n).

The trade-off: faster reads come at the cost of slower writes, since every index must be updated on each INSERT or UPDATE.

A B-tree index organizes keys in a balanced tree — a 3-level tree can find any row among billions in microseconds.

The B-Tree Index

A B-tree is a self-balancing tree data structure that keeps data sorted and enables searches, insertions, and deletions in logarithmic time. Most database indexes are built on B-trees.

B-tree indexes allow logarithmic search — millions of rows in microseconds

ACID Transactions

ACID transactions ensure database operations are reliable: either all succeed (commit) or none take effect (rollback). This prevents data corruption from crashes, network failures, or concurrent access.

  • Atomicity: All or nothing — the transaction completes fully or not at all
  • Consistency: Data follows defined rules and constraints at all times
  • Isolation: Concurrent transactions don't interfere with each other (MVCC)
  • Durability: Committed data persists even if the server crashes
ACID transactions guarantee atomicity: all operations commit together or roll back together — even on crash.
