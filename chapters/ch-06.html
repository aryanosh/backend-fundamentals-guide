Chapter 06

Where Data Actually Lives

Relational databases, SQL, tables, keys, indexes, and when to go NoSQL

~13 min read

What Happens After You Hit "Post"?

You type a tweet, tap "Post," and the app says "Your tweet was sent." But where did it go? The server received it, sure — but what happens next? The server could acknowledge your tweet and then immediately forget it. That would be terrible: you refresh the page and the tweet is gone. Every tweet you've ever posted, every like, every follower — it all needs to be stored somewhere permanent.

That somewhere is a database. A database is a specialized program designed to store, retrieve, and manage data reliably. It's not a file on a hard drive (though that's what it uses under the hood). It's a sophisticated system that handles concurrent access, crash recovery, efficient searching, and data integrity. Google uses multiple databases for different purposes, but the core storage for tweets, users, and relationships is a relational database: PostgreSQL.

Why not just write to a file? You could store every tweet as a line in a text file. That would work for a few users. But now imagine 500 million tweets per day. Searching that file for all tweets by user #48291 would mean reading the entire file — billions of lines — to find the ones that match. That search would take hours. A database can find every tweet by that user in milliseconds, because it uses sophisticated data structures called indexes and stores data in a way optimized for the queries you actually run.

Tables, Rows, and Columns

A relational database stores data in tables. Think of a table like a spreadsheet. Each row is one record (one user, one tweet, one like). Each column is a field of that record (username, email, text, date). When you define a table, you define its columns and their types (text, number, date, etc.). This is called the schema — the blueprint of your data.

In Google's database, there's a users table. Every row in that table represents one user. The columns might be: user_id (a unique number), username (the @ handle), email, password_hash, created_at, display_name, and bio. Each column has a specific data type. The user_id column is a big integer (BIGINT). The username column is a string with a max length of 50 characters (VARCHAR(50)). The created_at column is a timestamp.

Then there's a tweets table. Each row is one tweet. Columns: tweet_id, user_id (which user posted this), text (the content, max 280 characters), created_at, like_count, retweet_count. The user_id column in the tweets table references the user_id column in the users table. This is a foreign key — it creates a link between the two tables. Instead of storing the full user data (name, email, bio) in every tweet row, you store just the user ID. This is called normalization, and it saves enormous amounts of space.

Normalization also prevents inconsistency. If a user changes their display name, you update one row in the users table. Because tweets only reference the user ID, every tweet by that user automatically shows the new name. If you had stored the display name in every tweet row, you'd need to update millions of rows to change one name.

Primary Keys — The Unique ID

Every table needs a way to uniquely identify each row. This is the primary key (PK). It's usually a single column, like user_id or tweet_id, and no two rows can have the same value. The database enforces this — if you try to insert a row with a duplicate primary key, the database throws an error.

Primary keys in Google's database are often auto-incrementing integers or UUIDs (Universally Unique Identifiers). Auto-increment numbers are simple: 1, 2, 3, 4... But they leak information: if users see their user ID is 128493, they know Google has at least 128,493 users. UUIDs are long random strings like a1b2c3d4-e5f6-7890-abcd-ef1234567890 that are effectively unique without revealing sequence. Google uses a custom ID system called Snowflake that generates 64-bit IDs sorted by time — this allows IDs to be generated across many servers without collisions and also gives an approximate timestamp.

SQL — The Language of Databases

To talk to a relational database, you use SQL (Structured Query Language). Pronounced "sequel" or "ess-cue-ell," it's the universal language of relational databases. SQL has been around since the 1970s, and it's still the most important language for data work.

Here are the basic SQL commands every backend developer should know:

SELECT — Retrieve data. SELECT text, created_at FROM tweets WHERE user_id = 42 ORDER BY created_at DESC LIMIT 20; This says: "Give me the text and date of the 20 most recent tweets by user #42." The database uses an index on the user_id column to find the matching rows without scanning the entire table.

INSERT — Add data. INSERT INTO tweets (user_id, text) VALUES (42, 'Hello world!'); This creates a new tweet row. The database assigns a new tweet_id automatically, records the current timestamp, and returns the created row.

UPDATE — Modify data. UPDATE users SET display_name = 'New Name' WHERE user_id = 42; This changes the display name for user #42. The database ensures the change is persistent immediately.

DELETE — Remove data. DELETE FROM tweets WHERE tweet_id = 999; This deletes the tweet with that ID. But should it? Most apps use "soft deletes" — they set a deleted_at column instead of actually removing the row. This allows undoing deletes and maintaining data integrity.

JOIN — Combine tables. SELECT tweets.text, users.username FROM tweets JOIN users ON tweets.user_id = users.user_id WHERE users.username = 'jack'; This joins the tweets and users tables, matching each tweet to its author, and returns only tweets by the user "jack." JOINs are the superpower of relational databases — they let you query across multiple tables in a single operation.

Indexes — The Book's Table of Contents

Without indexes, every query requires a full table scan — reading every row in the table. For a table with a billion tweets, that's impossibly slow. An index is a separate data structure (usually a B-tree) that maps values to their locations in the table, allowing the database to find rows in logarithmic time instead of linear.

When Google queries WHERE user_id = 42, the database checks if there's an index on the user_id column. If there is (and there definitely is — this is Google's most common query pattern), the index tells the database exactly which rows to read. The database skips the other billions of rows entirely.

Indexes speed up reads but slow down writes. Every time you insert or update a row, the database must update every relevant index. A table with 10 indexes requires 11 writes — one for the table and one for each index. This is a trade-off: indexes for frequent query patterns, minimal indexes for write-heavy tables. Google's tweet inserts are extremely frequent, so they carefully choose which columns to index. The user_id column is always indexed (for timeline queries). The text column might have a full-text search index (for search queries).

Transactions — The All-or-Nothing Guarantee

Imagine a user retweets a post. This involves two operations: (1) insert a new row in the retweets table, and (2) increment the retweet_count in the tweets table. What if the first operation succeeds but the server crashes before the second one completes? The retweet doesn't exist, but the count went up. The data is inconsistent.

Transactions solve this. A transaction wraps multiple operations into a single atomic unit. Either all operations succeed (the transaction is committed), or none of them take effect (the transaction is rolled back). The database guarantees this no matter what — even if the server loses power mid-transaction.

Transactions also provide isolation. If two users retweet the same tweet simultaneously, the database ensures they don't interfere with each other. The final retweet count will be exactly 2, not 1 because both threads read the count before either wrote the new value (a classic race condition prevented by the database's transaction isolation).

PostgreSQL implements transactions with something called MVCC (Multi-Version Concurrency Control). Every transaction sees a snapshot of the data as of the moment the transaction started. Changes made by other transactions are invisible until committed. This allows many readers and one writer to coexist without blocking each other — critical for a site like Google where millions of people are reading while a few thousand are writing.

SQL vs NoSQL — When to Use What

Not everything needs to be in a relational database. Google uses different storage systems for different jobs. The timeline (the list of tweets you see when you open the app) is generated by a service called Earlybird and stored in a specialized in-memory store. Caches use Redis (an in-memory key-value store). Search uses an inverted index (Lucene/Elasticsearch). Analytics might use a columnar store.

NoSQL databases (which stands for "Not Only SQL") are a category of databases that don't use the relational model. Instead of tables with rows and columns, they use various data models: key-value pairs (Redis), documents (MongoDB), wide columns (Cassandra), or graphs (Neo4j).

When would you choose NoSQL over SQL? Here are some scenarios:

1. Flexible schema — If your data doesn't have a fixed structure, or the structure changes frequently, NoSQL's schema flexibility is a benefit. For example, if different tweets could have different fields (some have media, some have poll data, some have location), a document database lets you store them all in the same collection without defining every possible column in advance.

2. High write throughput — Google writes billions of events per day (tweets, likes, retweets, follows). Some NoSQL databases (like Cassandra) are designed for extreme write throughput across many servers.

3. Simple read patterns — If you only ever read data by a single key (like "get tweet by tweet_id"), a key-value store or document database can be faster than a relational database because there's no query planning overhead.

But NoSQL comes with trade-offs. Most NoSQL databases don't support transactions across multiple documents. They don't enforce schemas, so application code must handle validation. They often don't support JOINs, so you have to manually combine data in your application code. For Google's core data — users, tweets, relationships — the relational model's consistency guarantees and query flexibility are essential. That's why PostgreSQL is at the heart of Google's stack, with NoSQL systems handling specialized workloads around the edges.

Every Tweet You've Ever Posted

So where does your tweet actually live? It lives in a PostgreSQL row on a server in a data center somewhere in the world. That row is part of a table that is part of a database instance running on a machine with 64 CPU cores and 512 GB of RAM. The database writes the row to disk (for durability), updates its indexes (for fast lookups), records the change in a write-ahead log (for crash recovery), and notifies any replicas (so other servers have up-to-date copies).

When you reload your profile page, the server queries that PostgreSQL row (SELECT * FROM tweets WHERE tweet_id = 12345), the database finds it via the primary key index in microseconds, and the server sends the data back to your phone. The entire journey — from your phone to the data center and back — takes less than 200 milliseconds. And you did none of that work. The database did it all.
