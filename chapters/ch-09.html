Making Things Fast with Caching

Why waiting 200ms for a database query is too slow — and what to do about it

~12 min read

Cache Layers

Caching stores a copy of data somewhere fast so you don't fetch from the original slow source every time. Think of it like your kitchen pantry: instead of driving to the grocery store (database) for salt, you keep a jar in your cupboard (cache).

Browser Cache — HTTP headers like Cache-Control: max-age=3600 keep traffic off your servers. First line of defense.

Application Cache (Redis / Memcached) — In-memory key-value store for query results, session data, rate limit counters.

Database Cache — MySQL buffer pool, Postgres shared buffers. Databases cache in memory too.

Cache hierarchy: L1 in-memory (fastest) ? L2 Redis ? Database (slowest). Each miss falls through to the next layer

CDNs — Content Delivery Networks

A CDN is a network of servers distributed around the world. When someone in Tokyo visits a site hosted in New York, a CDN serves cached static assets from the nearest edge server, cutting load times from seconds to milliseconds.

CDNs cache static assets (images, CSS, JS, fonts) and increasingly API responses. They also provide DDoS protection by absorbing traffic across thousands of edge servers.

CDN edge servers around the world cache content and serve users from the nearest location, reducing latency

Cache Invalidation Strategies

Four strategies for keeping cache and database in sync, each with different trade-offs between consistency, latency, and complexity.

Four cache invalidation strategies: TTL, Write-Through, Write-Behind, and Cache-Aside

Cache Invalidation is Hard

Phil Karlton famously said: "There are only two hard things in computer science: cache invalidation and naming things." Once you cache data, the original can change. You need a strategy to remove or update stale data.

Three classic approaches: TTL expiration, write-through (sync), and write-behind (async)

Write Strategies

Write-Through: Write to cache and DB simultaneously. Cache stays fresh but write latency doubles.

Write-Behind: Write to cache immediately, then async to DB. Fast writes with risk of data loss on crash.

Write-Around: Bypass cache for writes. Good for write-once, read-rarely data.

Redis Example

Google uses Redis extensively for trending topics, timelines, rate limit counters, and session storage. When you open Google's Explore page, the computation result is cached in Redis with a short TTL. The next 9,999 users read the cached result in under a millisecond instead of triggering a full recomputation.
