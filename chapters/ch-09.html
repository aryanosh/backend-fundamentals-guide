Chapter 09

Making Things Fast with Caching

Why waiting 200ms for a database query is too slow — and what to do about it

~12 min read

What Is Caching?

Caching means storing a copy of data somewhere fast so you don't have to fetch it from the original slow source every time. Think of it like your kitchen pantry: instead of driving to the grocery store (database) every time you need salt, you keep a small jar of it in your cupboard (cache). It's the same salt — you just saved yourself a trip.

In backend systems, the "slow source" is usually a database that takes 50 to 500 milliseconds to answer a query. The "fast source" is something like Redis or an in-memory store that can answer in under a millisecond. When you serve millions of requests, shaving off those milliseconds is the difference between a snappy app and a slow one.

Caching is one of the most powerful performance tools in a backend engineer's toolbox. It's also one of the easiest to get wrong — because now you have to keep those copies in sync.

Types of Caches

1. Browser Cache

Your browser caches things automatically. Images, CSS files, JavaScript — when you visit a site, the browser says "I'll keep this for a while" so the next visit loads faster. The server tells the browser how long to cache via HTTP headers like Cache-Control: max-age=3600. This is the first line of defense: it keeps traffic off your servers entirely.

2. CDN (Content Delivery Network)

A CDN is a network of servers distributed around the world. When someone in Tokyo visits a site hosted in New York, a CDN serves cached static assets from a server in Tokyo. Companies like Cloudflare, Fastly, and Akamai specialize in this. For users far from your origin server, a CDN can cut load times from seconds to milliseconds.

3. Application Cache (Redis / Memcached)

This is where you store computed data in memory so subsequent requests don't recompute it. Redis is the most popular choice — it's an in-memory key-value store that can persist data to disk. You might cache:

  • Database query results (e.g., "all tweets for the Explore page")
  • Session data (user logged in? here's their info)
  • Rate limiter counters
  • Computed values (leaderboards, trending topics)

4. Database Cache

Databases cache too. MySQL keeps frequently queried rows in its buffer pool (in memory). Postgres has shared buffers. Even with a cache in front, the database itself uses memory to speed up reads. Tuning these buffers is a common DBA task.

Cache Invalidation: The Hardest Problem in Computer Science

Phil Karlton famously said: "There are only two hard things in computer science: cache invalidation and naming things." Here's why it's hard: once you cache data, the original data can change. If a user updates their profile, the cached copy is now stale. You need a strategy to either remove or update that stale data.

Two common approaches:

TTL (Time-To-Live)

You set a timer on cached data. After 60 seconds (or 5 minutes, or an hour), the cache entry expires and the next request fetches fresh data. This is simple and reliable. The downside: some users might see data that's up to 60 seconds old. For most apps, that's fine.

Event-Based Invalidation

When data changes, you actively delete or update the cache entry. For example, when a user posts a new tweet, you delete the cached "home timeline" so the next request rebuilds it. This gives you fresh data instantly, but it adds complexity — you need to know exactly which cache keys to invalidate.

Twitter's Approach

Twitter's trending topics are a textbook caching case. When you open the Explore page, Twitter needs to compute which topics are trending right now. That computation involves analyzing millions of tweets, counting hashtags, applying filters, and ranking results. Doing that for every single request would melt the database.

Here's what actually happens: the first person to open the Explore page triggers the computation. The result gets stored in Redis with a TTL of a few minutes. When 10,000 people open the Explore page in the next second, 9,999 of them read the cached result in under a millisecond. Only one person (or maybe zero, if a new computation is needed) triggers the expensive query.

Twitter uses Redis extensively — for timelines, trending topics, rate limit counters, and session storage. In fact, Twitter engineers are some of the OSS contributors behind Redis and Memcached because they've pushed these tools to their limits at massive scale.

When Not to Cache

Not everything should be cached. Real-time data like stock prices or live sports scores needs to be fresh. User account balances in a banking app should be read from the source of truth. Cache when stale data is acceptable; don't cache when accuracy is critical.

Also: caching adds operational complexity. You need to run Redis (or similar), configure eviction policies, handle cache misses gracefully, and monitor cache hit ratios. If your app is small, you might not need caching at all.

Key Takeaways

  • Caching stores a copy of data closer to the user or in faster storage to avoid slow queries
  • Browser cache and CDNs keep traffic off your servers entirely
  • Redis is the go-to in-memory cache for backend applications
  • TTL-based expiration is simple; event-based invalidation gives fresher data with more complexity
  • Cache invalidation is genuinely tricky — plan your strategy upfront
  • A cache hit ratio above 90% is generally considered healthy
