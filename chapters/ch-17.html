Chapter 17

The Learning Roadmap

A phase-by-phase plan to become a backend engineer

~20 min read

This Is the Plan

The engineers who built Twitter did not start knowing microservices. They started with the same fundamentals you are learning now. They learned one concept at a time, built messy projects, broke things, fixed them, and got better. There is no shortcut. But there is a clear path. This roadmap lays it out in five phases, with week estimates so you can pace yourself.

Stick to this sequence. Do not jump ahead. Each phase builds on the previous one. If you try to learn Docker before you know how to build a REST API, you will be confused. If you try to learn microservices before you understand databases, you will be building on sand.

Phase 1: Foundation (Weeks 1–6)

What to study: Pick one language — Python is the best choice for beginners — and learn its syntax cold. Variables, data types, conditionals, loops, functions, arrays/lists, dictionaries/maps, classes, file I/O, error handling. Work through a structured course or book. Do not skip exercises. You need the muscle memory of typing code every day.

Also learn the absolute basics of the terminal: how to navigate directories, run your scripts, and use a text editor (VS Code is fine). Learn Git fundamentals: init, add, commit, push, pull.

What to build: A command-line calculator. A to-do list manager that saves to a file. A temperature converter. A dice-rolling simulator. Simple projects that require you to use the syntax you learned.

What to avoid: Frameworks. Do not touch Django, Flask, Express, or any other web framework yet. Do not study algorithms and data structures obsessively — you just need to know arrays, objects, loops, and recursion at this point. Do not worry about performance. Do not try to build a startup. Type code. That is the only goal.

Twitter connection: This phase is where you build the foundation that every Twitter engineer started with. The Python you learn here is the same Python used in Twitter's data pipelines and analytics systems.

Phase 2: Databases & APIs (Weeks 7–12)

What to study: Learn SQL. Not ORMs, not query builders — raw SQL. SELECT, INSERT, UPDATE, DELETE, JOIN, GROUP BY, ORDER BY, WHERE clauses. Subqueries. Indexes (what they are and why they matter). Pick PostgreSQL as your database. It is the most powerful open-source relational database and the one you will encounter most in the industry.

Learn what an API is. HTTP methods (GET, POST, PUT, DELETE). Status codes (200, 201, 400, 401, 404, 500). Request and response structure. JSON. Then build a REST API using a minimal framework — Flask in Python, or Express in Node.js, or Sinatra in Ruby. Keep it simple. The API should return JSON and talk to your PostgreSQL database.

What to build: A simple blog API. Endpoints: create a post, get all posts, get one post, update a post, delete a post. Store posts in PostgreSQL. Write the SQL queries yourself. Test your API with curl or Postman.

What to avoid: ORMs (SQLAlchemy, Prisma, ActiveRecord). Authentication. Caching. Microservices. Deployment. These are distractions right now. You need to understand the raw layers before you add the abstractions.

Twitter connection: Twitter's API serves billions of requests per day. Every tweet, like, retweet, and follow is an API call backed by databases. The fundamentals you learn here are what Twitter's API is built on — just at a much larger scale.

Phase 3: Real World (Weeks 13–20)

What to study: Authentication and authorization. How passwords are hashed (bcrypt, argon2). How sessions work. JWTs (JSON Web Tokens) — what they are, how they are signed, how to verify them. Environment variables for configuration.

Learn migrations — how to evolve your database schema over time. Learn about background jobs (why you should not send emails during an HTTP request). Learn about logging and basic error handling in production.

Learn about MVC architecture. Separate your routes, your business logic, and your data access into different layers. Your codebase should be organized, not one giant file.

Start using a test framework. Write unit tests for your utility functions. Write integration tests for your API endpoints. Aim for at least some test coverage.

What to build: A user authentication system. A URL shortener. A note-taking app with user accounts. A project that requires you to handle user registration, login, protected routes, and data that belongs to specific users.

What to avoid: Kubernetes. Microservices. Event-driven architectures. Serverless. CI/CD pipelines. You will get to those. Not yet.

Twitter connection: Every feature at Twitter — from tweeting to DMs to bookmarks — involves user authentication, data ownership, and background processing. This phase teaches you to build those features at a small scale.

Phase 4: Production (Weeks 21–28)

What to study: Docker. Understand images, containers, Dockerfiles, docker-compose. Containerize your application so it runs the same everywhere.

Learn about cloud deployment. Create an AWS account. Learn about EC2 (virtual machines) and RDS (managed databases). Deploy your application to a real server that is accessible from the internet.

Set up a CI/CD pipeline. GitHub Actions is free and easy. Every push should run your tests. If they pass, deploy to your server.

Learn about load balancing, horizontal scaling, and database connection pooling (at a conceptual level). Learn about monitoring — tools like Sentry for error tracking and basic logging.

What to build: Take your application from Phase 3, containerize it, deploy it to AWS, and set up CI/CD. Your app should be running on a real domain/IP with automated deployments.

What to avoid: Kubernetes. Multi-region deployments. Infrastructure-as-code tools like Terraform. Microservices architecture. These are real skills but they require the foundation you build in this phase.

Twitter connection: Production engineering is what Twitter does every day. Every service is containerized, every deploy is automated, every system is monitored. This phase turns your "works on my machine" project into something that could handle real users.

Phase 5: Level Up (Ongoing)

What to study: This phase never ends. Here is what to explore next, in rough order:

  • Advanced databases: Indexing strategies, query optimization, read replicas, connection pooling. Learn about NoSQL databases like Redis (caching) and MongoDB (document storage).
  • Message queues: RabbitMQ, Kafka, or Redis streams. Understand pub/sub, event-driven architectures, and async processing.
  • Microservices: When and why to split a monolith. Service communication (HTTP, gRPC, message queues). Service discovery. The trade-offs.
  • Infrastructure-as-code: Terraform, Ansible, or Pulumi. Treat your infrastructure like code — versioned, reviewed, automated.
  • Observability: Logging, metrics, and tracing. The three pillars. Tools like Prometheus, Grafana, OpenTelemetry.
  • Security: OWASP Top 10, SQL injection prevention, XSS, CSRF, rate limiting, API security best practices.
  • Performance: Profiling, caching strategies, database optimization, CDN usage, connection pooling.

What to build: Start a larger project. A real application that you maintain for months. Contribute to open source. Write blog posts about what you learn. The act of teaching solidifies knowledge.

What to avoid: Getting paralyzed by choice. There are a thousand things you could learn. Pick one, learn it deeply, then move to the next. Do not obsess over the latest framework or tool.

Twitter connection: Even senior engineers at Twitter are in Phase 5. They constantly learn new tools, debug novel problems, and deepen their understanding of the systems they maintain. The learning never stops. That is what makes it a career, not just a job.

Final Advice

Learn the fundamentals deeply. No framework first — learn the language, the database, the protocol. Frameworks come and go. HTTP does not. SQL does not. The relational model does not. Linux does not. Git does not. Investing in these fundamentals pays dividends for your entire career.

Build things. Not tutorials — actual projects that you design, build, debug, and ship. You learn more from one project that you built yourself than from ten courses you followed along with. The bugs you fix are the lessons you keep.

Read code. Read the source code of the tools you use. Read PRs on open-source projects. Understanding how other people structure code is one of the fastest ways to improve.

Be patient. Backend engineering is deep and wide. Nobody masters it in a year. The goal is not to know everything — the goal is to be better than you were last week. Keep shipping. Keep learning. The rest takes care of itself.
