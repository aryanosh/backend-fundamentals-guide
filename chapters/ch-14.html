Chapter 14

Version Control with Git

Never lose your work — and collaborate without chaos

~14 min read

What is Git?

Git is a version control system. That is a fancy way of saying it tracks every change made to your code, forever. Every edit, every new file, every deletion — Git records it with a timestamp, a message explaining why the change was made, and the author's name. You can rewind to any point in history, compare versions, and branch off to try ideas without risking the main codebase.

Before Git, teams shared code by copying files over a network drive or emailing ZIP archives. It was a nightmare. Files got overwritten. Changes disappeared. People cried. Git, created by Linus Torvalds (the same person who created Linux), solved this for good.

How Git Works

A Git repository is a folder that Git is watching. You create one with git init. Inside that folder, Git creates a hidden .git directory that stores the entire history. Once initialized, you can start tracking changes.

The basic cycle is: edit files, git add to stage them, git commit to save a snapshot. Each commit is a permanent checkpoint with a unique hash. You write a commit message describing what you did and why. Good commit messages are short yet descriptive: "Fix login timeout on mobile" not "fixed stuff".

Your code lives in your local repository. When you are ready to share it, you git push to a remote repository (like GitHub, GitLab, or Bitbucket). Other people git pull to get your changes. This is how hundreds of engineers at Google work on the same codebase without stepping on each other's toes.

Branching and Merging

A branch is a parallel version of your code. The default branch is usually called main (or master). When you want to add a feature, you create a new branch off of main. You work there, making commits. Meanwhile, other engineers are working on their own branches. Nobody touches main directly.

When your feature is complete, you open a pull request (PR). A PR is a request to merge your branch into main. Other engineers review your code, leave comments, and suggest changes. Once it passes review and all tests pass, it gets merged. git merge combines the commits from your branch into main.

This is the branching strategy used by most professional teams: protect main, branch for everything, merge through PRs. It keeps the main branch stable and deployable at all times.

Essential Commands

git init — create a new repository. git add — stage changes for commit. git commit -m "message" — save a snapshot. git branch feature-x — create a branch. git checkout feature-x — switch to it. git merge feature-x — merge it into the current branch. git push — send commits to the remote. git pull — fetch and integrate remote changes. git log — view commit history. git status — see what has changed.

That is maybe a dozen commands. With those, you can handle virtually every situation that arises in professional software development.

Git at Google

Hundreds of engineers work on Google's codebase at the same time. Git is how they do it without overwriting each other's work. Frontend engineers work on the web app. Backend engineers work on the API. Data engineers work on analytics pipelines. All on different branches, all against the same repository.

When Google ships a new feature — say, bookmarks — the process looks like this: an engineer creates a branch called bookmarks-feature, writes the code, opens a PR, gets reviews from senior engineers, runs automated tests, and merges to main. From there, it goes through a deployment pipeline and eventually reaches users. Every step is tracked in Git. Every change has an author. Every decision has context.

If something breaks, git bisect helps find the exact commit that introduced the bug. If a feature is not ready, the branch stays open while main keeps moving forward. This is not a luxury — it is a necessity when hundreds of people are shipping code every day.
