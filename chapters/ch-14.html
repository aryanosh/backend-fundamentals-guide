Version Control with Git

Never lose your work — and collaborate without chaos

~14 min read

What is Version Control?

Git is a version control system that tracks every change made to your code, forever — with timestamps, messages, and authors. You can rewind to any point in history, compare versions, and branch off to try ideas without risking the main codebase.

  • git init — create a new repository in any folder
  • git add and git commit — stage and save snapshots of your work
  • A branch is a parallel version of your code for isolated development
Branches diverge from main, accumulate commits, and merge back — keeping main stable

Basic Git Commands

With a dozen commands you can handle virtually every situation in professional software development. The core cycle is: edit, stage, commit, push.

  • git status — see what has changed in your working directory
  • git log — view the commit history
  • git push and git pull — sync changes with a remote repository
  • git merge and git rebase — integrate changes from different branches
Merge creates a merge commit preserving fork history; rebase replays commits linearly

Pull Requests

A pull request (PR) is a request to merge your branch into main. Other engineers review your code, leave comments, and suggest changes. Once it passes review and all tests pass, it gets merged. This is the standard way teams collaborate — every change is reviewed, tested, and tracked before reaching production.

The standard PR workflow — each step is tracked, reviewed, and verified before reaching production

Merging

git merge combines commits from different branches into a single unified history. git rebase rewrites commit history for a cleaner linear graph, but should never be used on shared branches as it rewrites history and causes conflicts for other developers.
