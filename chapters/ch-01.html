How Machines Find Each Other

DNS, IP addresses, ports, and the handshake that starts everything

~10 min read

How Do Computers Even Find Each Other?

The DNS lookup: browser queries a resolver, which returns Google's IP address

Imagine you want to mail a letter to Google headquarters. You can't just write "Google" on an envelope — the postal service needs a street address. The internet works the exact same way. Every computer connected to the internet has a numeric address called an IP address.

An IP address looks like this: 104.244.42.65 (that's one of Google's real IP addresses). It's four numbers separated by dots, each ranging from 0 to 255. If you think of the internet as a giant city, IP addresses are like street addresses — they tell packets of data exactly where to go.

But humans are terrible at remembering numbers. You know google.com without thinking, but not 104.244.42.65. That's where DNS — the Domain Name System — comes in. DNS is the internet's phonebook. When you type google.com, your computer asks a DNS server "What's the IP?" and gets back the address it needs.

This all happens in milliseconds. Behind the scenes, there's a hierarchy: your computer checks its local cache, then asks a recursive resolver, which may ask a root server, then a TLD server (.com), then the authoritative name server for google.com.

Ports: The Apartment Number

IP address = building address, port = apartment number. Together they form a socket.

So now your browser has Google's IP address. But Google's server runs many things at once — a web server, an API server, a database, maybe an email server. How does your request reach the web server and not the database? This is where ports come in.

If the IP address is the building's street address, a port is the apartment number. The standard port for web traffic is port 80 (HTTP) and port 443 (HTTPS). When you visit https://google.com, your browser connects to port 443 on 104.244.42.65. The operating system sees that connection on port 443 and routes it to the web server listening there.

There are 65,535 possible ports. Ports 0-1023 are "well-known ports": 22 for SSH, 25 for SMTP, 53 for DNS, 80 for HTTP, 443 for HTTPS. Higher ports are used for temporary connections — your browser might use a random port like 54321 on its end, connecting to port 443 on Google's end. This combo is called a socket.

The TCP Three-Way Handshake

The TCP three-way handshake: SYN, SYN-ACK, ACK — three messages establish a reliable connection

Your browser knows the IP address and the port. Now it needs to actually open a connection. The internet is built on TCP — Transmission Control Protocol. TCP is like a phone call: before you start talking, you make sure the other person is there and ready to listen. This setup is called the three-way handshake.

Step 1: SYN — Your browser sends a packet with a SYN flag to Google's server. "Hey, I'd like to start a connection. Here's my starting sequence number."

Step 2: SYN-ACK — Google's server receives the SYN. If it's willing to accept the connection, it sends back SYN-ACK. "I received your request. Yes, I'm ready to talk. Here's my sequence number."

Step 3: ACK — Your browser sends back ACK: "Great, I got your acknowledgment. Connection is open. Let's send data." At this point, both sides agree the connection is established.

This entire handshake happens in the blink of an eye — typically under 100 milliseconds. But it happens every single time you visit a website.

The Google Example: What Happens When You Open the App

The full chain from tapping "Search" to seeing results — six steps, under 200ms

Let's trace the complete flow. You tap Google Search. Here's the conversation below the surface:

1. Your phone checks if it has Google's IP cached. If not, it queries DNS: "What's the IP for google.com?"
2. The resolver responds: "104.244.42.65."
3. Your phone picks a random source port (say 43210) and sends SYN to 104.244.42.65:443.
4. Google's load balancer receives the SYN and responds with SYN-ACK. Your phone sends ACK. Connection open.
5. A TLS handshake establishes encryption (HTTPS).
6. Your phone sends GET / HTTP/2 with headers. Google processes it and returns search results.

You never see any of this. Behind a single tap, there's an orchestrated conversation involving DNS servers, load balancers, and multiple protocol layers — all designed so data arrives reliably, in order, and securely.
