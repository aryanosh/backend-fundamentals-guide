Chapter 01

How Machines Find Each Other

DNS, IP addresses, ports, and the handshake that starts everything

~10 min read

How Do Computers Even Find Each Other?

Imagine you want to mail a letter to Google headquarters. You can't just write "Google" on an envelope and drop it in a mailbox — the postal service needs a street address. The internet works the exact same way. Every computer, phone, or server connected to the internet has a numeric address called an IP address (Internet Protocol address). Google's servers have one. Your phone has one. The laptop you're reading this on has one.

An IP address looks like this: 104.244.42.65 (that's one of Google's real IP addresses). It's four numbers separated by dots, each ranging from 0 to 255. If you think of the internet as a giant city, IP addresses are like street addresses — they tell packets of data exactly where to go. There are two versions in use today: IPv4 (the one with four numbers, like above) and IPv6 (a newer, longer format with eight groups of hex digits that looks like 2001:db8::1). IPv6 exists because we literally ran out of IPv4 addresses — there are more devices than available IPv4 addresses now.

But here's the problem: humans are terrible at remembering strings of numbers. You probably know "google.com" without thinking, but would you memorize 104.244.42.65? Probably not. That's where DNS — the Domain Name System — comes in. DNS is essentially the internet's phonebook. When you type google.com into your browser, your computer doesn't know where that is. So it calls up a DNS server (usually provided by your internet service provider or a public service like Google's 8.8.8.8) and asks, "Hey, what's the IP address for google.com?" The DNS server looks it up and replies: "It's 104.244.42.65." Your browser now has the address it needs.

This DNS lookup happens in milliseconds. Behind the scenes, there's a whole hierarchy: your computer first checks its local cache (has it looked up google.com recently?), then asks a recursive resolver, which might ask a root name server, then a TLD server (.com), then the authoritative name server for google.com. Each level narrows it down until the exact IP is found. It's like asking a receptionist, who asks a floor manager, who asks a department head, who finally points you to the right desk. All of this happens before a single byte of Google's actual content reaches your screen.

Ports: The Apartment Number

So now your browser has Google's IP address. Great. But Google's server is running lots of things at once — there's a web server, an API server, a database, maybe an email server. How does your browser's request reach specifically the web server and not the database? This is where ports come in.

If the IP address is the building's street address, a port is the apartment number. The standard port for web traffic is port 80 (HTTP — unencrypted web traffic) and port 443 (HTTPS — encrypted web traffic). When you visit https://google.com, your browser automatically connects to port 443 on 104.244.42.65. The server's operating system sees that incoming connection on port 443 and knows to route it to the web server software (like nginx or Apache) that's been configured to listen on that port.

There are 65,535 possible ports in total. Ports 0-1023 are "well-known ports" reserved for standard services: 22 for SSH, 25 for SMTP email, 53 for DNS, 80 for HTTP, 443 for HTTPS. Higher-numbered ports are used for temporary connections — when your browser opens a connection, it might use a random port like 54321 on its end, connecting to port 443 on Google's end. This is called a socket: the combination of IP address and port forms a unique endpoint (192.168.1.5:54321 → 104.244.42.65:443).

The TCP Three-Way Handshake

Your browser knows the IP address and the port. Now it needs to actually open a connection. The internet is built on a protocol called TCP — Transmission Control Protocol. TCP is like a phone call: before you start talking, you make sure the other person is there and ready to listen. This setup is called the three-way handshake, and it's one of the most fundamental concepts in all of networking.

Here's exactly what happens, step by step:

Step 1: SYN — Your browser sends a packet with a SYN (synchronize) flag to Google's server. This packet says, "Hey, I'd like to start a connection. Here's my starting sequence number." Think of it like picking up a phone and dialing. The packet includes a random sequence number (let's say 1000) that helps both sides keep track of what data has been sent and received.

Step 2: SYN-ACK — Google's server receives the SYN packet. If it's willing and able to accept the connection (its port 443 is open and listening), it sends back a SYN-ACK packet. This says, "I received your request. Yes, I'm ready to talk. Here's my starting sequence number (5000), and I acknowledge yours (1001)." The acknowledgment number is always the received sequence number plus one, because that +1 represents the next byte expected.

Step 3: ACK — Your browser sends back an ACK (acknowledgment) packet: "Great, I got your acknowledgment. Connection is open. Let's start sending data." At this point, both sides agree the connection is established, and your browser can finally send its HTTP request (things like "GET /home HTTP/1.1" with all the headers).

This entire handshake happens in the blink of an eye — typically under 100 milliseconds. But it happens every single time you visit a website. If Google has thousands of images and scripts, that could mean dozens of TCP connections opening simultaneously. Modern browsers and HTTP/2/3 optimize this with features like connection reuse (keep-alive) and multiplexing, but the fundamental three-way handshake is still there underneath.

The Google Example: What Happens When You Open the App

Let's trace this with a real-world example. You're on your phone and you tap Google Search. Here's the conversation that happens below the surface:

1. Your phone checks if it has Google's IP address cached from last time. If not, it sends a DNS query: "What's the IP for google.com?"
2. The DNS resolver (maybe your phone carrier's DNS, maybe 1.1.1.1) responds: "It's 104.244.42.65."
3. Your phone's networking stack picks a random source port (say 43210) and sends a SYN packet to 104.244.42.65:443.
4. Google's load balancer (a specialized server that distributes incoming traffic) receives the SYN. It responds with SYN-ACK.
5. Your phone sends ACK. The TCP connection is now open.
6. Over this connection, your phone sends an encrypted TLS handshake (HTTPS layer) to establish a secure channel.
7. Finally, your phone sends the actual HTTP request: GET / HTTP/2 with headers like User-Agent, Accept-Language, and Cookie.
8. Google's servers process the request and send back the search results — all within a few hundred milliseconds.

You never see any of this. You just see your search results load. But behind that single screen tap, there's a beautifully orchestrated conversation involving multiple servers, multiple protocol layers, and multiple handshakes — all designed so that data arrives reliably, in order, and without corruption.

Now that you understand how machines find each other and establish a connection, you're ready for the next chapter: what exactly gets sent over that connection. Because knowing how the pipe is built is only half the story — the real magic is in what flows through it.
