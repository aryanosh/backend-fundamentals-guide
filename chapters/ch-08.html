Chapter 08

Authentication & Authorization

Who you are, what you're allowed to do, and how the server knows you're not lying

~13 min read

Why Your Password Doesn't Travel With Every Request

When you log into Twitter, you type your username and password. The app sends them to the server. The server checks: "Is this the right password for this username?" If yes, the server says "OK, you're in." But here's the important part: after that login, the app never sends your password again. Not for loading your timeline. Not for posting a tweet. Not for changing your settings. Your password is used exactly once: at login.

Why? If the app sent your password with every request, anyone who intercepted even one of those requests would have your password forever. They could log in as you from any device, change your password, lock you out. The risk is enormous. Instead, after the server confirms your password is correct, it gives the app something else — a token — that proves "this request is authorized as user #42." The token is temporary. It expires after an hour, or a day, or whenever you log out. If someone steals the token, they only have access for a limited time, and you can invalidate it by logging out (which tells the server to blacklist that token).

This chapter covers two distinct concepts that beginners often mix up: authentication (AuthN) and authorization (AuthZ). Authentication is "who are you?" — verifying identity. Authorization is "what are you allowed to do?" — checking permissions. They work together but they are different problems solved with different tools.

Authentication — Proving Who You Are

Authentication answers the question: "Are you really who you claim to be?" The most common form is a password. You claim to be @jack. To prove it, you provide the password that only the real @jack should know. The server checks: does this password match what we have stored for @jack? If yes, you're authenticated.

But here's the thing: the server should never store your actual password. If Twitter's database was breached (it has been, in 2018), attackers would get access to every user's stored password if they were stored in plain text. That's catastrophic because people reuse passwords across sites. Instead, Twitter stores a hash of your password — the output of a one-way mathematical function that turns your password into a fixed-length string of gibberish.

password "hunter2" → hash function → "ab3d8f1c..."

The crucial property of hashing is that it's one-way. You can compute the hash from the password, but you cannot compute the password from the hash. When you log in, Twitter hashes your input and compares it to the stored hash. If they match, the password was correct. But even if the database is stolen, the attackers only get the hashes, not the passwords. They would have to try billions of password guesses to find one that produces the same hash — which is computationally expensive if the hash function is designed to be slow.

Twitter uses bcrypt, a hash function specifically designed for passwords. Unlike fast hashes like MD5 or SHA-256 (which are designed for file integrity checking and can compute billions of hashes per second), bcrypt is intentionally slow. It has a "cost factor" that determines how many rounds of computation are needed. A single bcrypt hash might take 100 milliseconds to compute. That's fast enough for a login attempt (you wait 100 ms, no big deal), but it makes brute-force attacks impractical — trying a billion password guesses would take 100 million seconds, which is over 3 years.

Bcrypt also automatically handles salting. A salt is a random value added to the password before hashing. Every user gets a different salt. This means even if two users have the same password (both use "hunter2"), their hashes will be completely different. This prevents rainbow table attacks — precomputed lists of hashes for common passwords. With salting, an attacker would need to generate a separate rainbow table for every salt, which is impractical.

The Login Flow — Step by Step

Let's trace what happens from the moment you tap "Log in" on Twitter to the moment you see your timeline:

1. The app collects your username and password and sends them to POST https://api.twitter.com/2/oauth2/token (or similar endpoint) over HTTPS. HTTPS encrypts the entire request so no one on the network can see your password.

2. The server receives the credentials. It looks up your username in the database. If the username doesn't exist, it returns a generic error: "Invalid username or password." (Notice it doesn't say which one is wrong — that would tell attackers whether a username exists.)

3. The server retrieves the stored bcrypt hash for your account. It hashes the password you provided using the same salt and cost factor. If the computed hash matches the stored hash, you're authenticated. If not, the server returns "Invalid username or password."

4. Once authenticated, the server creates a session or generates a token. In modern APIs, this is almost always a JWT (JSON Web Token). The JWT contains a payload that identifies you: your user ID, your username, and some metadata (when the token was issued, when it expires). The server signs this payload with a secret key that only the server knows.

5. The server returns the JWT to the app. The app stores it securely (in iOS Keychain, Android Keystore, or an HTTP-only cookie for web apps). Crucially, the app stores the token, not the password.

6. For every subsequent request, the app includes the JWT in the Authorization header: Authorization: Bearer eyJhbGciOiJIUzI1NiIs.... The server verifies the signature, checks the expiration, extracts the user ID from the payload, and knows: "This request is from user #42."

JWT — The JSON Token That Can't Be Forged

A JWT is not a random string. It's a carefully structured token with three parts, separated by dots: header.payload.signature. Each part is base64-encoded (not encrypted — anyone can decode it). The security comes from the signature, not from hiding the content.

The header says what algorithm was used to create the signature (usually HS256 for HMAC with SHA-256, or RS256 for RSA). The payload contains the claims — statements about the user and the token itself. Standard claims include sub (subject — the user ID), iat (issued at — when the token was created), exp (expiration — when the token becomes invalid), and iss (issuer — who created this token). Custom claims are common too: Twitter might include {"role": "admin"} or {"scope": "read write"}.

The signature is what prevents tampering. The server creates it by taking the base64-encoded header, concatenating it with a dot and the base64-encoded payload, and hashing the whole thing with a secret key that only the server knows. If anyone modifies the header or payload (like changing "user_id": 42 to "user_id": 1 to impersonate an admin), the signature verification will fail because the computed signature won't match the one on the token.

This is the beauty of JWTs: they are stateless. The server doesn't need to store the token in a database. Every request includes everything the server needs to verify identity. The server just checks: "Is the signature valid? Is the token expired?" If both checks pass, the request is authenticated. This makes JWTs perfect for distributed systems where multiple servers handle requests — any server with the secret key can verify any token, without a central session store.

But JWTs also have a downside: they can't be revoked (before expiration) without maintaining a blacklist, which defeats the purpose of statelessness. If a user reports their phone stolen, Twitter can't invalidate the token until it expires. That's why JWTs usually have short expiration times (15 minutes to 1 hour). For longer sessions, a refresh token is used: a longer-lived token stored securely on the server that can be exchanged for new JWTs without requiring re-login. If a refresh token is compromised, it can be revoked because it's stored on the server.

Authorization — What You're Allowed to Do

Authorization happens after authentication. The server knows who you are (authenticated via JWT). Now it checks: "Is user #42 allowed to delete tweet #999?" If the tweet belongs to user #42, the answer is yes. If it belongs to user #7, the answer is no.

Authorization can be as simple as "can the owner of this resource access it?" or as complex as role-based access control (RBAC) with dozens of permissions. Twitter uses a combination of approaches:

Ownership checks — You can edit your own tweets, not someone else's. The server checks: tweet.author_id == current_user.id.

Role-based permissions — Twitter employees have admin roles that allow them to moderate content, suspend users, or view analytics. The JWT might include a role claim: {"role": "admin"}. The server checks: if current_user.role == 'admin': allow.

Scope-based permissions — When a third-party app asks to access your Twitter account via OAuth, it requests specific scopes: tweet.read, tweet.write, users.read. The JWT issued to that app only includes the granted scopes. If the app tries to post a tweet but only has tweet.read scope, the server rejects the request.

Authorization is typically implemented as middleware — a function that runs before the request handler. The middleware checks the JWT, extracts the user ID and role, and attaches them to the request object. Then the route handler checks permissions before performing the action. If authorization fails, the server returns 403 Forbidden (not 401 Unauthorized — that's for authentication failures).

OAuth — Log in With Google

You've used "Log in with Google" or "Log in with Apple" a hundred times. This is OAuth 2.0, the industry standard protocol for delegated authorization. OAuth allows you to grant one application access to your data on another application without sharing your password.

Here's how it works when you tap "Log in with Google" on Twitter:

1. Twitter redirects you to Google's authorization page. The URL includes Twitter's client ID (identifying Twitter to Google), a redirect URI (where to send you after authorization), and the scopes Twitter is requesting (like "see your email address" and "see your name").

2. You log in to Google (if not already logged in) and see a page: "Twitter wants to access your email and profile info. Allow?" You click Allow.

3. Google redirects you back to Twitter with a temporary authorization code in the URL.

4. Twitter's server takes that code and sends it to Google's server (server-to-server, not visible to you) along with Twitter's client secret. Google verifies the code and returns an access token (and optionally a JWT ID token).

5. Twitter uses the access token to call Google's API: GET https://www.googleapis.com/oauth2/v2/userinfo with the token in the header. Google returns your email and name.

6. Twitter finds or creates an account with that email and logs you in. Twitter never sees your Google password.

The key insight: Twitter never handles your Google password. Twitter only gets a limited-scope, revocable token. If you later go to your Google security settings and revoke Twitter's access, Twitter's token stops working immediately — even if you never change your Google password.

This is the fundamental pattern of OAuth: delegation without credential sharing. It's why you can log into hundreds of sites with your Google account without trusting every one of those sites with your Google password.

Putting It All Together — The Secured Request

Let's follow one complete secured request: you post a tweet.

1. You type "Hello world!" and tap Post. The app retrieves the JWT from secure storage.

2. The app sends POST https://api.twitter.com/2/tweets with Authorization: Bearer eyJhbG... and {"text": "Hello world!"} in the body.

3. The server's auth middleware intercepts the request. It extracts the JWT from the Authorization header. It verifies the signature against the server's secret key. It checks the exp claim — is the token still valid? It decodes the payload to get user_id: 42. All of this happens in a few milliseconds, usually in a shared library used by every route.

4. The middleware attaches the user object ({id: 42, username: "jack", role: "user"}) to the request and passes it to the route handler.

5. The route handler checks authorization: "Is user #42 allowed to post tweets?" If they're not suspended and they have the tweet.write scope, yes. It then creates the tweet in the database.

6. The server returns the created tweet with status 201 Created. The app displays it in the timeline.

The entire flow — login, token issuance, token verification, authorization — happens without your password ever leaving the initial login request. That's the fundamental security pattern of the modern web.
