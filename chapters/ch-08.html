Authentication & Authorization

Who you are, what you're allowed to do, and how the server knows you're not lying

~13 min read

What is Authentication?

Authentication (AuthN) is the process of verifying identity — answering "who are you?" When you log into an app, you prove you are who you claim to be, typically with a password, biometric, or social login.

Authentication differs from authorization (AuthZ). AuthN verifies identity; AuthZ checks permissions. They work together but solve different problems with different tools.

  • Password-based — user provides a secret only they should know. The server hashes and compares.
  • Token-based — after login, the server issues a signed token (JWT) that proves identity for subsequent requests.
  • OAuth / Social — delegate authentication to a trusted provider (Google, Apple) without sharing credentials.
JWT = header.payload.signature — anyone can read it, but only the server with the secret can sign it

Login Flow

When you log into an app, your credentials travel through a secure pipeline: from client to server, where they are validated, and if correct, a token is returned for subsequent requests.

The password is used exactly once — after that, a signed JWT proves your identity without ever exposing your secret again.

The login flow: credentials are exchanged for a signed token, which proves identity on every subsequent request

JWT Structure

A JWT is not a random string. It has three dot-separated parts: header.payload.signature. Each part is base64-encoded (not encrypted), so anyone can decode the content. Security comes from the signature, not from hiding the data.

The header specifies the signing algorithm (HS256 or RS256). The payload contains claims — user ID, role, expiration time, and issuer. The signature prevents tampering: the server hashes the header + payload with a secret key. If the data is modified, the signature becomes invalid. This makes JWTs stateless — no database lookup needed to verify identity.

Session vs JWT

Two main strategies exist for maintaining user identity after login: sessions and JWTs. Each approach makes different trade-offs between simplicity, scalability, and security.

Sessions are stateful — the server stores session data. JWTs are stateless — the token itself contains all the information needed to identify the user.

Sessions are stateful (server stores data); JWTs are stateless (token contains everything)

OAuth 2.0

OAuth 2.0 is the industry standard for delegated authorization — letting one application access your data on another application without sharing your password. It is the protocol behind every "Log in with Google" or "Log in with Apple" button.

The core pattern is delegation without credential sharing: you authorize an app to act on your behalf, and the app receives a limited-scope, revocable token. Your password is never exposed to the third party.

OAuth 2.0 delegates identity without sharing credentials — the app never sees your password

Password Hashing

Authentication answers "are you who you claim to be?" The most common proof is a password. But the server must never store the actual password — only a hash of it, produced by a one-way mathematical function.

A good hash function like bcrypt is intentionally slow and includes a unique salt per user. This makes brute-force attacks computationally expensive even if the database is breached. The password is hashed, compared, and never stored in plain text.

Hashing is one-way and slow (good for passwords); encryption is reversible (good for data at rest)

Encryption vs Hashing

Hashing and encryption serve fundamentally different purposes. Hashing is a one-way function: you can turn a password into a hash, but you cannot reverse the hash back to the password. It is designed to be slow and includes a unique salt per user. This makes it ideal for storing passwords.

Encryption is a two-way function: data encrypted with a key can be decrypted with the same (symmetric) or a related (asymmetric) key. Encryption is useful for data at rest (encrypting database fields) or data in transit (TLS). But encryption should never be used for passwords — if the key is stolen, all passwords are exposed.
