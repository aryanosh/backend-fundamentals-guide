Chapter 16

Deployment & DevOps

Getting your code from your laptop to the world

~15 min read

From Laptop to Production

You have written your code. It works on your machine. Now how do you get it to the world? That is what deployment is about. And it is not as simple as copying files to a server. Modern deployment is automated, tested, and repeatable. It involves containers, pipelines, and cloud infrastructure.

Docker: It Works on My Machine

The most famous phrase in software engineering is "It works on my machine." Docker exists to kill that phrase. Docker packages your application and everything it needs — the runtime, libraries, environment variables, config files — into a container. A container is like a lightweight virtual machine. It runs the same way on your laptop, your teammate's laptop, and the production server.

A Dockerfile describes how to build the container. Here is a simple one for a Node.js app:

FROM node:18
WORKDIR /app
COPY package.json .
RUN npm install
COPY . .
EXPOSE 3000
CMD ["node", "server.js"]

You build the image with docker build -t myapp . and run it with docker run -p 3000:3000 myapp. The app is now running in an isolated environment that behaves identically everywhere.

Twitter uses containers extensively. Each microservice runs in its own container. When a new version needs to be deployed, a fresh container is built, tested, and rolled out. If something fails, the old container is still running and traffic can be switched back instantly.

CI/CD: Automate Everything

CI/CD stands for Continuous Integration and Continuous Deployment. It is the pipeline that runs every time you push code. The pipeline builds your application, runs your tests, and if everything passes, deploys to production. All automatically. No humans needed after the push.

Every push to Twitter's repository triggers their CI/CD pipeline. The pipeline builds each microservice, runs the full test suite — thousands of unit tests, hundreds of integration tests, dozens of E2E tests — and if all pass, deploys the services that changed. The entire process takes minutes. If a test fails, the pipeline stops and notifies the team. No broken code ever reaches production without someone knowing about it first.

This is the standard for professional backend engineering. You set it up once, and it runs forever, catching problems that would otherwise slip through.

Cloud Basics

You do not buy physical servers anymore. You rent them from cloud providers like AWS, Google Cloud, or Azure. You spin up a virtual machine (called an EC2 instance in AWS), install your software, and point your domain at it. When you need more capacity, you add more instances. When traffic drops, you scale down. You pay only for what you use.

Cloud services also provide managed databases, load balancers, DNS, content delivery networks, and dozens of other tools. The learning curve is steep, but the flexibility is unmatched. Twitter runs on its own infrastructure, but most startups and even large companies rely on cloud providers.

Environment Variables

Hardcoding configuration in your source code is a security risk and a maintenance nightmare. Instead, use environment variables. Database URLs, API keys, secret tokens — they all go into environment variables. In development, you set them in a .env file. In production, the CI/CD pipeline injects them when the container starts.

This keeps secrets out of your codebase. Your code reads process.env.DATABASE_URL (or the equivalent in your language) and the actual value comes from the environment. Same code, different configurations for development, staging, and production.

Deployment at Twitter

Twitter deploys code hundreds of times per day. Every deploy goes through an automated pipeline that catches problems before they reach users. The process is: an engineer merges a PR to main, the CI/CD pipeline detects the change, builds the affected services, runs the test suite, deploys to a staging environment for final validation, and then rolls out to production gradually. If error rates spike, the pipeline automatically rolls back. Users experience zero downtime.

This is the end goal of DevOps: merging code and having it safely running in production minutes later, with every safety check automated along the way.
