Chapter 07

How Systems Talk to Each Other

REST APIs, JSON, URL conventions, GraphQL, and the art of fetching just what you need

~12 min read

Your Phone Doesn't Talk to the Database

Here's a common misunderstanding among beginners: when you open the Twitter app on your phone, your phone doesn't connect directly to Twitter's PostgreSQL database. That would be a security nightmare and a performance disaster. Your phone would need the database credentials (which could be stolen), it could run arbitrary SQL queries (like DROP TABLE users), and it would need to maintain a persistent connection (battery drain). No, there's a layer in between: the API.

An API (Application Programming Interface) is a middleman. It's a server that sits between the client (your phone or browser) and the database. The client sends a request to the API. The API validates the request, checks permissions, applies business logic (like "is this tweet under 280 characters?"), queries the database if needed, and returns a response. The client never touches the database directly.

Twitter's API is a REST API (Representational State Transfer). REST isn't a protocol or a library — it's a set of design principles. It's a way of organizing your API so that it feels consistent, predictable, and intuitive. When you learn what GET /tweets does, you can guess what GET /users does, even if you've never seen it before. That consistency is the superpower of REST.

Let's look at the actual REST APIs that Twitter uses, because they're the perfect example of how real-world APIs are designed.

The Anatomy of a REST Request

Every API request has a few fundamental parts. Think of it like sending a letter: you need an address (URL), a method (what you want to do), headers (metadata), and sometimes a body (the content).

The URL (Uniform Resource Locator) identifies the resource you want to interact with. Twitter's API URL might look like https://api.twitter.com/2/tweets. The /2/ is the API version — when Twitter makes breaking changes, they increment this number so old apps don't break. The /tweets part identifies the resource (the collection of tweets).

The HTTP method (also called verb) tells the server what action you want to take. There are four main methods you'll use every day:

GET — Retrieve data. This is the most common method. When your timeline loads, the app sends GET https://api.twitter.com/2/tweets?ids=123,456. GET requests should never change anything on the server. They're "safe" — you can send the same GET request a million times and nothing changes (except maybe the server load).

POST — Create something new. When you post a tweet, the app sends POST https://api.twitter.com/2/tweets with a JSON body containing {"text": "Hello world!"}. The server creates a new tweet and returns it with a 201 Created status.

PUT — Update or replace something. If you edit your bio, the app might send PUT https://api.twitter.com/2/users/42 with your new profile data in the body. PUT typically replaces the entire resource — you send the full new version.

DELETE — Remove something. When you delete a tweet, the app sends DELETE https://api.twitter.com/2/tweets/123. Twitter returns 204 No Content to confirm it's gone.

The headers contain metadata about the request. The most important header is Authorization, which carries your authentication token (we'll dive deep into that in the next chapter). Other headers include Content-Type (what format the body is in, usually application/json), Accept (what format you want the response in), and User-Agent (what app is making the request).

The body is the payload — the actual data being sent. For POST and PUT requests, the body contains the data you're creating or updating. For GET and DELETE requests, the body is usually empty. Responses also have a body — the data you requested — almost always formatted as JSON.

JSON — The Lingua Franca of APIs

JSON (JavaScript Object Notation) is the most common format for API data. It's lightweight, human-readable, and supported by every programming language. A JSON response from Twitter's API might look like this:

{ "data": { "id": "123456", "text": "Hello world!", "author_id": "42", "created_at": "2024-01-15T10:30:00Z" } }

JSON has a few simple rules: objects are wrapped in curly braces {} and contain key-value pairs. Arrays are wrapped in square brackets [] and contain ordered lists. Strings are in double quotes. Numbers don't have quotes. Booleans are true or false. Null is null. That's it — you can represent any data structure with just these rules.

Twitter's API responses are wrapped in a data key. Why? This allows Twitter to include metadata alongside the response without mixing it with the data. For example, a response might include a meta object with pagination info: {"data": [...], "meta": {"next_token": "abc123", "result_count": 10}}. The client reads the data to render tweets and the meta to know if there are more tweets to load.

URL Naming Conventions — The Art of the Path

Good API design follows conventions that make the API self-documenting. Here are the key rules that Twitter and most major APIs follow:

Use nouns, not verbs. The URL should name the resource, not the action. /tweets is good. /getTweets or /createTweet is bad — the HTTP method already communicates the action. If you find yourself putting verbs in URLs, you're probably doing something wrong.

Use plural nouns for collections. /tweets (collection), /tweets/123 (specific member). /users, /users/42. This is consistent and predictable.

Nest related resources. /users/42/tweets means "tweets by user 42." /tweets/123/likes means "likes on tweet 123." Nesting shows ownership — the relationship between resources. But don't nest too deep. Two levels is usually the max; deeper nesting suggests a design problem.

Use query parameters for filtering, sorting, and pagination. /tweets?limit=20&offset=40 means "give me tweets 41-60." /tweets?author_id=42 means "filter to tweets by this author." Query parameters modify the request without changing the resource identity.

Version your API. Twitter uses /2/ in the path. This allows them to introduce breaking changes without breaking existing apps. Old apps continue to use v1.1, new apps use v2. Without versioning, every change to the API would break every client immediately.

Pagination — When One Response Isn't Enough

Twitter can't return every tweet in a single response. That would be gigabytes of data. Instead, APIs split results into pages. The client requests one page at a time, and the response includes a token or cursor for getting the next page.

There are two main pagination approaches. Offset pagination uses ?offset=40&limit=20 — skip 40 items, return 20. This is simple but slow on large datasets (the database has to count through 40 items before it can return the 20 you want). Cursor-based pagination uses opaque tokens: ?cursor=abc123. The token encodes where you left off, and the database can jump directly there using an index. Twitter uses cursor-based pagination for its API. The response includes a next_token in the meta object. When the client scrolls to the bottom, it requests the next page with that token. No token means no more pages.

Cursor-based pagination is more robust because it handles data changes gracefully. If a new tweet is inserted while you're paginating, offset pagination might show you the same tweet twice (offset shifts), but cursor pagination always gives you the next batch after the last item you saw.

GraphQL — A Different Philosophy

REST is not the only way to design APIs. GraphQL, developed by Facebook (now Meta), offers a different approach. Instead of multiple endpoints returning fixed data shapes, GraphQL exposes a single endpoint and lets the client specify exactly what data it needs.

With a REST API, loading a user's profile with their recent tweets might require two requests: GET /users/42 and GET /users/42/tweets. With GraphQL, you send one query: { user(id: 42) { name, bio, tweets(limit: 5) { text, created_at } } }. The server returns exactly the fields you asked for — no more, no less.

This solves a common REST problem called over-fetching (getting more data than you need) and under-fetching (not getting enough data, requiring multiple requests). Twitter's mobile app uses GraphQL for some parts of its API. When you load a tweet detail page, the app sends a GraphQL query that asks for the tweet text, the author's name and avatar, the like and retweet counts, and the first 20 replies — all in one request.

But GraphQL has trade-offs. It shifts complexity from the client to the server. The server must parse the query, resolve each field, and handle authorization for every field individually. A malicious client could request a deeply nested query that crashes the server (this is called a DoS attack via GraphQL). REST is simpler to cache, simpler to monitor, and simpler to reason about. For most APIs, starting with REST is the right call. GraphQL is a powerful tool for specific problems — when you have multiple clients that each need different data shapes from the same server.

The Real Twitter API in Action

When the Twitter mobile app loads your timeline, here's exactly what happens at the API level:

1. The app calls GET https://api.twitter.com/2/timelines/home.json with your auth token in the header. Twitter's API v2 uses a different endpoint structure for timelines, but the principles are the same.

2. The server receives the request. It extracts your user ID from the auth token. It queries a service called Earlybird (Twitter's in-memory timeline service) which has pre-computed your timeline — a list of tweet IDs from people you follow, ranked by relevance.

3. The server then fetches the full tweet data (text, author, media, engagement counts) for those IDs, often from a cache called TwiterCache (yes, that's the real name). If a tweet isn't cached, it falls back to PostgreSQL.

4. The server assembles the JSON response: {"data": [{tweet}, {tweet}, ...], "meta": {"next_token": "xyz"}}. It includes only the fields the app requested (via GraphQL or REST query parameters).

5. The response is sent back over HTTPS, compressed with gzip, and your phone renders the timeline. The entire API call takes 150-400 milliseconds.

6. When you scroll to the bottom, the app sends the same request with ?cursor=xyz to get the next page of results. This continues until there are no more tweets to load.

The beauty of this API design is that it decouples the client from the server. Twitter can change the backend entirely — replace Earlybird with a new service, migrate from PostgreSQL to something else — and as long as the API contract (the endpoints and their response shapes) stays the same, the mobile app keeps working without updates. This is the architectural principle of separation of concerns applied at the system level.
