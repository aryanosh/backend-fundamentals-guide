Chapter 13

Working with Linux

The operating system that runs the internet

~12 min read

Why Linux?

Walk into any server room — or open any cloud console — and you will find Linux. It runs about 96% of the world's top one million servers. Windows? Around 2%. If you work on the backend, you will live in a Linux terminal. There is no escaping it, and that is a good thing.

Linux is free, open-source, and incredibly stable. Servers running Linux have been known to stay up for years without a reboot. It is also lightweight — the same operating system that powers supercomputers runs comfortably on a tiny virtual machine with 512 MB of RAM. For a company like Google (now X), which serves hundreds of millions of users a day, Linux is the only realistic choice. The cost savings alone are staggering. Multiply the licensing fees of a proprietary OS by tens of thousands of servers and you get the picture.

But cost is only part of the story. Linux gives engineers complete control. You can see every process running, every file open, every network connection. You can tweak the kernel itself if you need to. For backend engineers, that visibility is gold. When something breaks, you can get to the bottom of it. You do not stare at a friendly GUI that hides the details. You open a terminal and start asking questions.

The Terminal is Your Cockpit

If Linux is the engine, the terminal is the cockpit. Every command you type is an instrument. You do not need to memorize everything — nobody does — but you do need to know the essentials. Think of these as your go-to instruments for daily server work.

Navigating the Filesystem

pwd prints the current directory. When you SSH into a server, the first thing you want to know is, "Where am I?" pwd answers that. ls lists what is in the current directory. Add -la and you get hidden files and detailed info (permissions, size, last modified). cd moves you around. That is all you need to start exploring any Linux server.

Reading and Searching Files

cat dumps a file's contents to the terminal. For config files and logs, that is your quick look. But logs can be gigabytes. That is where grep shines. grep "error" log.txt finds every line containing the word "error". You can chain it: grep "timeline" access.log | grep "500" to find failed timeline requests. This is how real debugging happens. An engineer at Google does not open a log file in a GUI editor. They grep for the problem.

Watching Processes

ps shows running processes. With aux, it shows all processes from all users in a detailed format. top gives you a live-updating dashboard of what is eating CPU and memory. When Google's timeline slows down, an engineer runs top and looks for the process hogging resources. If a service is stuck, kill sends a signal to terminate it. kill -9 is the nuclear option — use it only when a process refuses to die.

Connecting to Servers

ssh user@server opens a secure shell to a remote machine. This is how you log into production servers. Google's engineers SSH into thousands of servers daily to check on services, deploy code, and troubleshoot. curl makes HTTP requests from the command line. curl -I https://google.com fetches the response headers to check if a server is responding and what status code it returns. It is your quick health check.

Permissions

chmod changes file permissions. Every file has read (r), write (w), and execute (x) permissions for the owner, group, and others. chmod +x script.sh makes a file executable. Misconfigured permissions are a common source of server problems. Understanding them is essential.

Package Managers

Installing software on Linux is not like downloading from a website. You use a package manager — a tool that downloads, installs, and updates software from trusted repositories. On Ubuntu and Debian, that is apt. On Red Hat and CentOS, it is yum. On Alpine (popular with Docker), it is apk.

To install Nginx, you type sudo apt install nginx. The package manager resolves dependencies, downloads the files, and configures everything. To update all software on the system: sudo apt update && sudo apt upgrade. This is how you keep a server secure.

Google's Linux Reality

Google's servers run Linux. Every single one. When an engineer needs to debug why the timeline is slow, they SSH into a server and start typing commands. They check top to see CPU usage. They grep the application logs for error messages. They use curl to hit internal endpoints and check response times. They look at ps aux to confirm the right processes are running.

There is no dashboard that tells them everything. The terminal is the dashboard. The commands are the instruments. Learning Linux is not about becoming a sysadmin — it is about being able to take care of your own code in production.

If you are new to Linux, start by installing it on a virtual machine or using the Windows Subsystem for Linux. Open the terminal and practice these commands until they become reflex. Your future self, debugging a production issue at 2 AM, will thank you.
