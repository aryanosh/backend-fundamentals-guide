Chapter 10

Security Essentials

Defending your backend against the internet's worst

~15 min read

Why Security Matters

Every day, automated bots scan the internet looking for vulnerable servers. They try SQL injection on every input field. They probe for exposed endpoints. They attempt to log in with common passwords. If your backend has a single unprotected endpoint or a query that concatenates user input directly into SQL, you will get compromised.

Security isn't a feature you add at the end — it's a mindset you build into every line of code. This chapter covers the most common attacks and the patterns that stop them.

SQL Injection

SQL injection is the most famous database attack. It works like this: instead of typing a normal username, an attacker types SQL commands into a form field. If your code naively plugs that input into a query string, the database executes the attacker's SQL.

-- Vulnerable code — never do this
query = "SELECT * FROM users WHERE username = '" + userInput + "'"

-- Attacker types: admin' OR '1'='1
-- Query becomes: SELECT * FROM users WHERE username = 'admin' OR '1'='1'
-- Returns every user in the database

The fix is simple and non-negotiable: parameterized queries (also called prepared statements). Instead of building SQL strings, you send the query template separately from the data:

-- Safe code
query = "SELECT * FROM users WHERE username = ?"
db.execute(query, [userInput])

The database treats the user input as data, not as executable SQL. Even if the attacker types admin' OR '1'='1, it becomes a literal string to match against — not part of the query logic. Every modern database driver supports parameterized queries. There is never a reason to concatenate SQL.

XSS (Cross-Site Scripting)

XSS happens when an attacker injects JavaScript into a page that other users view. For example, if Google allowed HTML in tweets, someone could post <script>alert('hacked')</script> and every visitor would run that script. Real attacks steal cookies, redirect users to phishing sites, or perform actions as the logged-in user.

Defense: Sanitize all user-generated content before displaying it. Escape HTML characters (< → &lt;, > → &gt;). Use frameworks that auto-escape (React, Vue, modern template engines). Never use innerHTML with user data. Set the Content-Security-Policy HTTP header to restrict what scripts can run.

CSRF (Cross-Site Request Forgery)

CSRF tricks a logged-in user into performing an action they didn't intend. Imagine you're logged into Google and visit a malicious site. That site's page contains a hidden form that posts to google.com/tweet with your content. Because you're authenticated, the browser sends your session cookie and the tweet goes through.

Defense: Use anti-CSRF tokens — a random value embedded in each form that the server validates. The malicious site can't guess this token because it's different for every session. Modern frameworks like Rails, Django, and Spring Security include CSRF protection by default. Also, set SameSite=Strict or SameSite=Lax on your cookies to prevent them from being sent on cross-site requests.

Rate Limiting

Rate limiting restricts how many requests a client can make in a given time window. Without it, an attacker can brute-force passwords, scrape your entire database via API calls, or DDoS your server. Google limits the Google API to a certain number of requests per 15-minute window per user. If you exceed it, you get a 429 Too Many Requests response.

Common rate-limiting strategies include token bucket, sliding window, and fixed window counters (often stored in Redis). Even a simple limit of 100 requests per minute per IP address will stop most automated attacks.

HTTPS Everywhere

HTTPS encrypts traffic between the client and your server. Without it, anyone on the same Wi-Fi network (coffee shop, airport, hotel) can read everything you send and receive. Passwords, API keys, private messages — all in plain text. HTTPS uses TLS (formerly SSL) to encrypt the connection. Browsers now mark HTTP sites as "Not Secure." There's no excuse not to use HTTPS — services like Let's Encrypt provide free TLS certificates.

OWASP Top 10

The Open Web Application Security Project (OWASP) publishes a Top 10 list of the most critical web application security risks. Every backend developer should be familiar with it. As of the latest edition, the top risks include:

  • Broken Access Control — users accessing data they shouldn't
  • Cryptographic Failures — not using HTTPS, weak passwords
  • Injection — SQL, NoSQL, OS command injection
  • Insecure Design — missing security controls in the architecture
  • Security Misconfiguration — default passwords, verbose error messages

Read through the full OWASP Top 10 at least once. Bookmark it. Refer to it when designing new features.

Google's Security Reality

Google gets attacked by bots and hackers constantly. Every day, automated scripts try logging in with leaked passwords from other breaches (credential stuffing). Attackers probe for XSS vulnerabilities in tweet text and profile bios. Scrapers attempt to pull user data through the API faster than allowed. Google's security team mitigates these with rate limiting, input sanitization, CSRF tokens on every form, mandatory HTTPS, and aggressive bot detection. The lesson: security is never "done." It's an ongoing process of monitoring, patching, and improving.

Key Takeaways

  • Parameterized queries prevent SQL injection — never concatenate user input into SQL
  • Sanitize output to prevent XSS; auto-escaping frameworks help enormously
  • CSRF tokens and SameSite cookies protect against cross-site request forgery
  • Rate limiting prevents brute force and abuse; Redis is great for storing counters
  • HTTPS with TLS (Let's Encrypt is free) is mandatory for all production traffic
  • The OWASP Top 10 is your security checklist — review it regularly
  • Security is a process, not a one-time fix
