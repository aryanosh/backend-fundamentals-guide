Security Essentials

Defending your backend against the internet's worst

~15 min read

SQL Injection

SQL injection is the most famous database attack. Instead of typing a normal username, an attacker types SQL commands into a form field. If your code naively plugs that input into a query string, the database executes the attacker's SQL.

The fix is simple and non-negotiable: parameterized queries (also called prepared statements). The database treats input as data, not as executable SQL.

String concatenation vs parameterized queries: the difference between a data breach and a safe query

Cross-Site Scripting (XSS)

XSS happens when an attacker injects JavaScript into a page that other users view. Real attacks steal cookies, redirect users to phishing sites, or perform actions as the logged-in user.

Defense: Sanitize all user-generated content. Escape HTML characters. Use frameworks that auto-escape (React, Vue). Never use innerHTML with user data. Set Content-Security-Policy headers.

Three types of XSS: Stored (persistent), Reflected (non-persistent), and DOM-based (client-side only)

Cross-Site Request Forgery (CSRF)

CSRF tricks a logged-in user into performing an action they didn't intend. A malicious site contains a hidden form that posts to your server. Because the user is authenticated, the browser sends their session cookie and the action goes through.

Defense: Use anti-CSRF tokens — a random value embedded in each form that the server validates. Set SameSite=Strict or SameSite=Lax on cookies to prevent cross-site sending.

CSRF exploits the user's existing session. Tokens and SameSite cookies prevent unauthorized cross-site requests

Rate Limiting

Rate limiting restricts how many requests a client can make in a given time window. Without it, attackers can brute-force passwords, scrape databases, or DDoS your server.

Common strategies include token bucket, sliding window, and fixed window counters (often stored in Redis). A limit of 100 requests per minute per IP stops most automated attacks.

The token bucket algorithm: each request consumes a token, tokens refill over time, requests denied when bucket is empty

HTTPS & TLS

HTTPS encrypts traffic between the client and your server. Without it, anyone on the same network can read passwords, API keys, and private messages. Use Let's Encrypt for free TLS certificates. Browsers now mark HTTP sites as "Not Secure." There is no excuse not to use HTTPS.

Security Headers

Content-Security-Policy — Restricts what scripts, styles, and resources can load. Prevents XSS even if a script tag slips through.

Strict-Transport-Security — Forces browsers to always use HTTPS. Prevents downgrade attacks.

X-Frame-Options — Prevents your site from being embedded in an iframe on another domain (clickjacking protection).

X-Content-Type-Options — Prevents MIME-type sniffing.

Input Validation

Never trust user input. Validate on the server side — client-side validation is for UX, not security. Whitelist allowed values, reject everything else. Check types, lengths, ranges, and formats. Sanitize inputs to remove dangerous characters.

Dependency Security

Your dependencies are attack surface. Use tools like npm audit, pip audit, or Dependabot to find vulnerabilities. Pin versions and review changes before updating. Remove unused dependencies. A single compromised package can bring down your entire system.

Defense in Depth

No single security measure is enough. Layer your defenses: HTTPS + CSP + CSRF tokens + rate limiting + input validation + parameterized queries + dependency audits. If one layer fails, the next one catches it. Security is a mindset, not a checklist.
