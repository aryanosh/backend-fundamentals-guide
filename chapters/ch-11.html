Chapter 11

Architecture Patterns

Monoliths, microservices, MVC, and event-driven design — and when to use each

~14 min read

No "Right" Architecture — Only Tradeoffs

Architecture isn't about picking the trendiest pattern. It's about understanding tradeoffs. A monolith is simple but can become unwieldy at scale. Microservices offer flexibility but add operational complexity. Event-driven systems decouple services but make debugging harder. The best architecture is the one that solves your current problem without overcomplicating things.

Monolith: Start Here

A monolith is a single application that handles everything — user interface, business logic, database access, authentication, email sending, everything. It runs as one process, deploys as one unit, and is usually one codebase. Monoliths are simple to build, test, and deploy. They're the default choice for startups, small teams, and early-stage products.

The downsides come with scale. As the codebase grows, it becomes harder to reason about. A tiny change to one feature can break something unrelated. Deploy times increase. Teams step on each other's toes. But here's the secret: many successful companies ran monoliths for years before splitting them up.

Microservices: Split When It Hurts

Microservices break your application into smaller, independent services that communicate over the network (usually via HTTP or message queues). Each service owns its data and can be developed, deployed, and scaled independently.

You might have a Users Service (handles signup, login, profiles), a Tweets Service (stores and serves tweets), a Timeline Service (computes your home feed), and a Search Service (indexes and queries tweets). Each has its own database and its own API.

The Good

  • Independent scaling: If the Timeline Service needs 100 instances but Users Service needs only 2, you can scale them separately
  • Team autonomy: Different teams own different services and can deploy independently
  • Isolation: A bug in one service doesn't take down the whole system
  • Technology diversity: Use the best tool for each job — Postgres for user data, Redis for caching, Elasticsearch for search

The Bad

  • Operational complexity: You now manage N services instead of 1 — monitoring, logging, deployment, networking
  • Distributed systems are hard: Network latency, partial failures, data consistency, retries, timeouts
  • Testing is harder: You need to test interactions between services
  • Data duplication: Each service may need its own copy of data that another service owns

MVC: Model-View-Controller

MVC is a design pattern that separates your code into three layers:

  • Model: The data and business logic — represents real-world things like User, Tweet, Like. Handles database queries and validation.
  • View: What the user sees — HTML templates, JSON responses, or mobile app screens. In modern backends, this is often an API response rather than rendered HTML.
  • Controller: The glue — receives HTTP requests, asks the Model for data, and passes it to the View. Think of it as the waiter who takes your order and brings your food.

MVC isn't a full system architecture — it's a way to organize code within a single service. Frameworks like Rails (Ruby), Django (Python), and Spring Boot (Java) are heavily MVC-oriented. Even if you don't use a strict MVC framework, separating concerns into data, presentation, and logic layers is a good practice.

Event-Driven Architecture

In an event-driven system, services communicate by publishing and consuming events. When something happens (a user signs up, a tweet is posted, a like is recorded), the service publishes an event to a message queue (like RabbitMQ, Kafka, or Amazon SQS). Other services consume those events and react accordingly.

Example: When a user posts a tweet, the Tweets Service publishes a "tweet_created" event. The Timeline Service consumes it and adds the tweet to followers' home timelines. The Search Service consumes it and indexes the tweet. The Notification Service consumes it and sends push notifications to followers.

Event-driven systems are highly decoupled — services don't need to know about each other. They just emit and listen to events. The downside: eventual consistency. When you post a tweet, it might take a few seconds to appear in followers' timelines. Debugging is also harder because the flow of data is spread across multiple services and queues.

Google's Architecture Evolution

Google started as a monolith — a single Ruby on Rails application with a single MySQL database. It worked fine for thousands of users. But as Google grew to millions and then hundreds of millions of users, the monolith couldn't keep up. The database was getting hammered. Deployments took 30 minutes. A bug in any feature risked taking down the whole site.

Google gradually evolved into a microservices architecture. They broke the timeline into its own service (now called Timelines Service, built on Redis clusters). Search became its own service using Java and Lucene. The API layer was separated. User graph management (who follows whom) became its own service. Today, Google runs hundreds of services, each with its own team, codebase, and deployment pipeline.

The lesson? Google didn't start with microservices. They started simple and split things apart as the pain of the monolith exceeded the pain of distributed systems. That's the right approach for almost everyone.

Key Takeaways

  • Start with a monolith — it's simpler and faster to build
  • Migrate to microservices only when the monolith's pain (deploy time, team coordination, scaling) outweighs distributed system complexity
  • MVC helps organize code within a service by separating data, logic, and presentation
  • Event-driven architecture decouples services via message queues, at the cost of eventual consistency and debugging difficulty
  • Google evolved from monolith to microservices as they scaled — they didn't start there
  • There is no perfect architecture, only tradeoffs
