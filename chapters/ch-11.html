Architecture Patterns

Monoliths, microservices, MVC, and event-driven design — and when to use each

~14 min read

Monolithic Architecture

A monolithic architecture packs everything — UI, business logic, data access, database — into a single deployable unit. It's the default choice for startups and small teams because it's simple to build, test, and deploy.

As the codebase grows, the monolith becomes harder to reason about. A tiny change in one feature can break something unrelated, and deployments get riskier over time.

  • Single codebase: One repository, one deployment pipeline
  • Shared database: All features access the same data store
  • Simple scaling: Scale the entire application — not individual components

Microservices Architecture

Microservices break your application into smaller, independent services that communicate over the network. Each service owns its data and can be developed, deployed, and scaled independently by separate teams.

This flexibility comes with operational complexity — you now manage N services instead of 1, with network latency, partial failures, and data consistency challenges.

  • Independent scaling: Scale only the services that need it
  • Team autonomy: Different teams own and deploy different services
  • Fault isolation: A bug in one service doesn't take down the whole system

Monolith vs Microservices

Monolith bundles everything into one unit; microservices split into independent services with their own data and deployment pipelines

Event-Driven Architecture

In an event-driven system, services communicate by publishing and consuming events via a message queue. When something happens (a user signs up, a tweet is posted), the service publishes an event. Other services consume that event and react — without knowing about each other.

This decoupling comes at the cost of eventual consistency and harder debugging, since data flow is spread across multiple services and queues.

In the pub/sub pattern, a single event published to the bus is consumed by multiple independent subscribers

Message Queues

Message queues provide reliable asynchronous communication between services. A producer sends a message to a queue, and one of several workers picks it up for processing — called the competing consumers pattern. This enables high throughput and fault tolerance.

Common systems include RabbitMQ, Apache Kafka, Amazon SQS, and Redis Streams. Queues handle traffic spikes, decouple producers from consumers, and provide retry/dead-letter mechanisms for failed messages.

In the competing consumers pattern, workers pull messages from a shared queue and each message is processed by exactly one worker

Sync vs Async Communication

Synchronous calls block the caller; asynchronous communication via queues allows services to operate independently

Which Pattern to Choose?

Start with a monolith — it's simpler and faster to build. Split into microservices only when the monolith's pain (deploy time, team coordination, scaling bottlenecks) outweighs the complexity of distributed systems. Use event-driven patterns when services need to react to changes without tight coupling. Message queues are ideal for reliable async processing and traffic buffering. There is no perfect architecture — only tradeoffs that evolve as your system grows.
