Chapter 12

Networking Essentials

How data travels across the internet — and how to handle millions of connections

~13 min read

How the Internet Works (The 30-Second Version)

When you type google.com into your browser, your computer sends a request across the internet to Google's server. That request passes through your home router, your ISP's network, undersea cables, and multiple routers before reaching Google's data center. Google's server processes the request and sends a response back along the same path. This all happens in a few hundred milliseconds.

But how does your browser know where to find Google's server? And how do millions of people use Google simultaneously? That's what networking protocols and infrastructure handle.

The OSI Model (Simplified)

The OSI model breaks networking into 7 layers. As a backend engineer, you mostly care about three of them:

Application Layer (Layer 7)

This is where your code lives. HTTP, WebSocket, gRPC — these are application-layer protocols. When your browser sends an HTTP request to GET /api/tweets, it's using the application layer. You don't worry about how the data gets there — you just use the protocol's request/response format.

Transport Layer (Layer 4)

This layer handles end-to-end communication between two computers. Two main protocols:

  • TCP: Reliable, ordered delivery. Used by HTTP, WebSocket, SSH, email. If a packet is lost, TCP retransmits it. This guarantees that the data arrives intact, but adds latency.
  • UDP: Fast, no guarantees. Used by video streaming, gaming, DNS lookups. If a packet is lost, it's not retransmitted — the video might glitch for a frame, but that's better than buffering.

Network Layer (Layer 3)

This layer handles routing packets across the internet. The key protocol is IP (Internet Protocol) — the addressing system that gives every device an IP address. Routers use IP addresses to figure out where to send packets. Think of it like the postal system: the network layer handles the address and routing, while the transport layer ensures the package arrives intact.

WebSockets vs HTTP

HTTP follows a request-response pattern: the client asks, the server answers, then the connection closes. That's fine for loading a web page or fetching data, but what about real-time features like Google's live feed? Polling (the client asking "anything new?" every second) wastes resources.

WebSocket is a protocol that maintains an open, persistent connection between client and server. Both sides can send messages at any time — no polling needed. Google uses WebSockets (via a library called Google Streaming API) to push new tweets to your timeline in real time. When a new tweet appears without refreshing the page, that's WebSocket at work.

Load Balancing

No single server can handle all of Google's traffic. The solution: run many identical servers and distribute requests across them using a load balancer. The load balancer sits in front of your servers and acts as a traffic cop.

Round-Robin

The simplest strategy. The load balancer sends the first request to Server A, the second to Server B, the third to Server C, then loops back to Server A. It's fair and simple, but it doesn't account for server load — one server might be handling slow requests while another is idle.

Least Connections

Sends each new request to the server with the fewest active connections. This is smarter — if Server A is handling 50 long-running requests and Server B is handling 5 short ones, the next request goes to Server B. This works well when request processing times vary.

IP Hash

Computes a hash of the client's IP address to consistently route the same client to the same server. This is useful when you need session persistence (affinity) — for example, if the server stores temporary user data in memory. The downside: if a server goes down, those users' sessions are lost.

How Google Scales

Google doesn't run on one server. They have thousands. A load balancer sits in front, distributing your request to the least busy server. Google uses a mix of hardware load balancers (F5) and software (their own Twemproxy, plus HAProxy and Nginx). Behind the load balancer, services are further distributed by topic — a tweet request goes to the Tweets Service cluster, a timeline request goes to the Timeline Service cluster, and so on.

Google also uses anycast routing, which lets multiple data centers around the world announce the same IP address. Your request is automatically routed to the nearest data center. That's why Google loads fast whether you're in New York, Tokyo, or London.

Key Takeaways

  • The OSI model has 7 layers; as a backend engineer, focus on Application (HTTP/WebSocket), Transport (TCP/UDP), and Network (IP)
  • TCP guarantees delivery with retransmission; UDP is faster but lossy
  • WebSocket maintains a persistent connection for real-time communication, unlike HTTP's request-response cycle
  • Load balancers distribute traffic across multiple servers to scale horizontally and provide fault tolerance
  • Round-robin is simple; least connections is smarter for variable workloads; IP hash provides session persistence
  • Google uses multiple load balancing layers and anycast routing to serve hundreds of millions of users globally
