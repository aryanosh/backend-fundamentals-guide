Networking Essentials

How data travels across the internet — and how to handle millions of connections

The OSI Model

The OSI model breaks networking into 7 layers. As a backend engineer, you mostly care about the top three (Application, Presentation, Session) for the code you write, and the bottom four (Transport, Network, Data Link, Physical) for understanding how data moves across the wire. Each layer serves the layer above it, abstracting away lower-level details so you can focus on your protocol without worrying about electrical signals or routing tables.

TCP vs UDP

TCP (Transmission Control Protocol) guarantees reliable, ordered delivery. If a packet is lost, TCP retransmits it. This ensures data arrives intact but adds latency. Used by HTTP, WebSocket, SSH, and email.

UDP (User Datagram Protocol) is fast with no guarantees. Packets may be lost, arrive out of order, or be duplicated — the application handles it. Used by video streaming, gaming, DNS lookups, and real-time communications where speed matters more than perfect delivery.

  • TCP: Connection-oriented, guaranteed delivery, in-order, higher latency
  • UDP: Connectionless, no guarantees, low overhead, no ordering
  • Choose TCP when data integrity matters; choose UDP when speed matters and occasional loss is acceptable

WebSocket Upgrade

WebSocket upgrades an HTTP connection to a persistent full-duplex channel — both sides can send messages anytime without polling

Load Balancers

No single server can handle all traffic. Load balancers sit in front of your servers and distribute incoming requests across multiple backend instances. They provide fault tolerance (if a server fails, traffic routes to healthy ones) and enable horizontal scaling (add more servers to handle more load).

  • Round-Robin: Simple fair distribution — cycles through servers regardless of load
  • Least Connections: Sends requests to the server with the fewest active connections
  • IP Hash: Routes the same client to the same server for session persistence
A load balancer distributes incoming requests across multiple backend servers. If a server fails, traffic is routed to the remaining healthy servers.

HTTP/2 & HTTP/3

HTTP/1.1 opened one connection per request — loading a page with 100 assets meant 100 connections. HTTP/2 introduced multiplexing: multiple requests share a single TCP connection, with header compression and server push. This dramatically reduced latency for resource-heavy pages.

HTTP/3 goes further by replacing TCP with QUIC (built on UDP). QUIC eliminates head-of-line blocking (where one lost packet stalls all streams) and reduces connection setup time. HTTP/3 is already used by Google, YouTube, Facebook, and most major websites.

  • HTTP/1.1: One request per connection, text-based, simple
  • HTTP/2: Multiplexed streams, header compression, server push over TCP
  • HTTP/3: QUIC over UDP, no head-of-line blocking, 0-RTT handshake
