Chapter 05

Memory, Threads & Processes

How computers juggle millions of things at once — and why your phone doesn't explode

~12 min read

You Can't Serve Millions With One Pair of Hands

Imagine you work at a restaurant. It's lunch rush — 200 hungry customers. You are the only server. You take an order, walk to the kitchen, wait for the food, bring it back, take the next order, walk to the kitchen, wait... It doesn't matter how fast you walk. You can only do one thing at a time, and every time you wait for the kitchen, you're useless to every other customer. The restaurant loses money. Customers leave.

Now imagine the restaurant hires 20 servers. Each server handles their own set of tables. When one server is waiting at the kitchen, the other 19 are still taking orders, clearing tables, and bringing food. The restaurant serves way more customers. This is the core idea behind threads. A single-threaded server is that one overwhelmed waiter. A multi-threaded server is the restaurant with 20 waiters — except the waiters can also share information with each other instantly.

When millions of users load Twitter simultaneously, one server can't handle them one at a time. That would mean user #10,000 has to wait for users 1 through 9,999 to finish before they even get a response. Nobody would wait that long. The answer is threads: the server spawns multiple threads, each handling a different user's request at the same time (or at least appearing to).

But before we dive deep into threads, we need to understand the stage they perform on: memory. Memory is where everything lives while the program runs — every variable, every user session, every cached tweet. How memory is organized determines what threads can and can't do.

RAM — The Computer's Desk

RAM (Random Access Memory) is your computer's short-term memory. Think of it as a giant desk where you spread out papers while working. The desk is big but not infinite — and everything on it disappears when you leave (when the power shuts off). Disks (SSDs, hard drives) are like filing cabinets: slower to access but permanent.

When you open the Twitter app on your phone, the operating system loads the Twitter server code from the filing cabinet (disk) onto the desk (RAM). Every variable, every connection, every piece of data the server is currently working with lives in RAM. Accessing RAM is fast — a few nanoseconds — compared to disk (a few milliseconds, a million times slower). This is why servers try to keep hot data in RAM and only write to disk when absolutely necessary.

A typical server has maybe 64 GB of RAM. At Twitter's scale, that's precious. Every byte of memory used by one request is a byte not available for another. That's why memory management — the art of allocating memory when needed and freeing it when done — is a fundamental backend skill.

Stack vs Heap — Two Kinds of Memory

When your program runs, memory is divided into two main regions: the stack and the heap. They serve different purposes and behave very differently. Understanding the difference is one of those things that separates developers who just write code from developers who understand what their code actually does.

The stack is like a stack of sticky notes. When a function is called, a new sticky note is placed on top. That note contains the function's local variables: a counter, a temporary string, a pointer to some data. When the function finishes, the sticky note is thrown away. Everything is neatly ordered — the last note placed is the first one removed (Last In, First Out, or LIFO). The stack is small (typically a few megabytes per thread) because it's only meant for temporary, short-lived data. The size of everything on the stack must be known in advance — you can't put something on the stack if you don't know how big it will be when the program starts.

The heap is the opposite. The heap is a giant open field where anything can be placed at any time, in any order. If your code needs to store a tweet text that could be 1 character or 280 characters, that goes on the heap because the size isn't fixed. If you need to share data between threads, it goes on the heap because the stack is private to each thread. The heap is large (you can use most of your 64 GB of RAM if needed), but it's slower to allocate and must be explicitly freed (or garbage-collected).

A simple rule: local variables with fixed sizes live on the stack. Everything else lives on the heap. When you write let user = fetchUser(id), the pointer to the user object lives on the stack, but the actual user data lives on the heap.

Processes — The Fortress

A process is a program in execution. It's the fortress that contains everything the program needs: its own memory space, its own file handles, its own network connections. Processes are isolated from each other — if one process crashes, it doesn't take down other processes. This is why your operating system can run Chrome, Spotify, and your terminal at the same time without them interfering with each other.

When Twitter starts its backend server, the operating system creates a process. That process gets its own chunk of RAM, its own entry in the OS process table, and its own security context. If the Twitter server crashes (a bug causes a null pointer exception, for example), only that process dies. The rest of the system keeps running.

The downside of this isolation is that processes cannot directly share memory. If Process A wants to send data to Process B, they must use inter-process communication (IPC): pipes, sockets, or shared memory segments explicitly set up by the OS. This is cumbersome and relatively slow.

Threads — The Workers Inside the Fortress

A thread is the smallest unit of execution within a process. A process can have one thread (single-threaded) or many threads (multi-threaded). All threads in the same process share the same memory space — they can all read and write the same variables, the same heap data, the same file handles. This is both the power and the danger of threads.

Here's a concrete Twitter scenario. The server starts a main thread that listens for incoming connections. When a new request arrives — say, "load my timeline" — the main thread hands that work to a worker thread from a pool. Thread #1 takes it. Thread #1 accesses the shared cache (heap) to check if the timeline is already cached. It's not, so it queries the database. While Thread #1 is waiting for the database to respond (blocked on I/O), the operating system lets Thread #2 handle a different request. Thread #2 might be handling a "post a tweet" request. It also accesses the shared cache. Thread #2 writes the new tweet to the cache. Now Thread #1 wakes up with the timeline data and returns it. Thread #3 handles another request. All three threads share the same cache, same database connection pool, same in-memory data structures.

This sharing is what makes threads so efficient. Without sharing, every request would need to load its own copy of the cache, its own database connections — a huge waste of memory. But sharing also introduces a classic problem: race conditions. If Thread #1 reads a value while Thread #2 writes to the same location, what does Thread #1 see? The old value? The new value? Something in between? This unpredictability leads to bugs that are notoriously hard to reproduce and fix. The solution is synchronization primitives like mutexes (mutual exclusion locks), which allow only one thread at a time to access a critical section of code. It's like a bathroom key — only one person can use the bathroom at a time, everyone else waits.

Blocking vs Non-Blocking

When a thread reads from a database or writes to a file, it usually has to wait for the operation to complete. This waiting is called blocking. The thread is "blocked" — it can't do anything else until the database responds. In a thread-per-request model (like early Java servers), each request gets its own thread. If there are 1000 concurrent requests, there are 1000 threads, even if 990 of them are just waiting for the database. This wastes memory because each thread's stack takes up space (usually 1-2 MB).

Non-blocking (or asynchronous) programming turns this around. Instead of assigning one thread per request, a small number of threads handle many requests. When a thread would normally block waiting for the database, it instead registers a callback ("when the database responds, call this function") and immediately moves on to handle another request. Node.js is famous for this model: a single thread can handle thousands of concurrent connections because it never blocks — it always moves on to the next task while waiting for I/O.

Think of it like a fast-food cashier. Blocking model: the cashier takes your order, walks to the kitchen, waits for the burger, brings it back, and only then takes the next order. Non-blocking model: the cashier takes your order, hands a slip to the kitchen, immediately takes the next order, hands another slip, takes another order, and when the first burger is ready, calls out your number. The cashier is never idle. This is event-driven programming, and it's how modern high-performance servers handle massive concurrency.

Concurrency vs Parallelism

These two terms are often used interchangeably, but they mean different things. Concurrency is about dealing with many things at once. Parallelism is about doing many things at once. The distinction matters because it shapes how you design your system.

Concurrency is like a juggler keeping multiple balls in the air. At any given moment, the juggler is only touching one ball, but they switch between balls so quickly that all balls stay in motion. A single-core CPU can achieve concurrency by rapidly switching between threads (this is called time-slicing). Each thread gets a tiny slice of CPU time, then the OS switches to the next thread. From the outside, it looks like everything is happening simultaneously.

Parallelism, on the other hand, requires multiple CPU cores. If the server has a 16-core CPU, it can literally execute 16 threads at the exact same moment — one per core. This is parallelism. Twitter's servers use parallelism to handle massive traffic: each CPU core handles different requests simultaneously, while concurrency allows each core to juggle many requests even when there are more requests than cores.

How much parallelism does Twitter actually need? A single server might handle 50,000 concurrent WebSocket connections (streaming tweets in real time). Most of those connections are idle — the user is just waiting for new tweets to appear. With non-blocking I/O, a server with 8 CPU cores and a modest amount of RAM can handle all 50,000 connections without breaking a sweat. The threads are concurrent (they share time on the cores) but not parallel (there are only 8 cores for 50,000 connections). The magic is that threads spend almost no time actively computing — they're almost always waiting for network data — so concurrency is sufficient.

Putting It All Together — The Twitter Server at Work

Let's walk through a concrete example. You open Twitter and your timeline loads. Here's what happens at the process/thread/memory level:

1. The main thread (listening on port 443) accepts your TCP connection. It passes the connection to a worker thread from the thread pool.

2. Worker thread #7 starts processing your request. It allocates a small stack frame for the request handler function. Local variables (like the request object pointer, a counter for retries) live on this stack.

3. The thread needs to check your auth token. It reads the token validation cache from the shared heap. If the token is valid and cached, great — no database call needed. If not, it reads from the database.

4. The thread queries the database for your timeline tweets. While waiting for the database to respond, does it block? That depends on the server architecture. In a traditional threaded server, yes — thread #7 waits. In a non-blocking server, thread #7 registers a callback and immediately handles another request while the database works.

5. The database returns results. Thread #7 (or whatever thread picks up the response) deserializes the data, builds the JSON response, and sends it back to your phone over the network connection.

6. Thread #7 returns to the thread pool, ready for the next request. Its stack memory is cleared. Any heap memory it allocated (for the response object, the parsed data) is garbage-collected or freed.

This cycle — accept, process, respond, recycle — happens millions of times per day on Twitter's servers. Each request is a tiny blip: allocate some memory, do some work, free the memory, move on. The art of backend engineering is making this cycle as efficient as possible: minimize heap allocations (they're slow), avoid blocking (wasted time), and synchronize access to shared data (prevent race conditions).

Why This Matters for Your Code

You might think: "I'm just building a simple API. Do I really need to understand threads and memory?" The answer is yes, because every backend framework you use operates on these principles. When you use Django, it spawns threads (or processes) to handle requests. When you use Node.js, it runs an event loop with a single thread. When you use Go, it uses lightweight "goroutines" that are even cheaper than threads. When you use Java with Spring, it manages a thread pool.

The concepts — stack vs heap, blocking vs non-blocking, concurrency vs parallelism — are the same regardless of language or framework. They're the physics of computation. Understanding them means you can predict how your application will behave under load before it crashes. You'll know why your memory usage keeps growing (heap allocations not freed), why your server stops responding (thread pool exhausted), or why your app slows down (blocking I/O).

In the next chapter, we'll take these threads and send them to the database — because data doesn't just live in RAM. It lives on disk, in tables, in indexes, and in carefully structured rows.
