Memory, Threads & Processes

How computers juggle millions of things at once — and why your phone doesn't explode

~12 min read

Stack vs Heap Memory

When your program runs, memory is divided into two main regions: the stack and the heap. The stack is fast and ordered (LIFO), but small and fixed-size; the heap is large and flexible, but slower to allocate.

  • Stack: local variables with fixed sizes, per-thread, fast allocation — automatically freed when the function returns
  • Heap: dynamic data, shared across threads, slower allocation — must be garbage-collected or explicitly freed
  • Pointers live on the stack; the actual data they point to lives on the heap
Stack frames grow and shrink with function calls (LIFO). Heap blocks persist until freed — shared across threads.

Processes vs Threads

A process is a program in execution with its own isolated memory space, file handles, and network connections. A thread is a unit of execution within a process — all threads share the same memory space, enabling fast communication but risking race conditions.

  • Processes: isolated, crash-safe, must use IPC to share data — higher overhead to create and manage
  • Threads: shared memory, lightweight, prone to race conditions — synchronized with mutexes (mutual exclusion locks)
  • Mutexes: allow only one thread at a time into a critical section, preventing data corruption
A process contains a shared heap and multiple threads, each with its own stack. Threads share memory; processes are isolated.

The Event Loop

The event loop is the heart of asynchronous runtimes like Node.js. A single thread cycles through phases — timers, I/O polling, checks, and close callbacks — processing events as they arrive without blocking on any single operation.

The event loop processes one phase at a time, never blocking

Blocking vs Non-Blocking I/O

Blocking I/O wastes thread time waiting for operations to complete — one thread per connection, most threads idle. Non-blocking I/O uses an event loop with callbacks, letting a single thread handle thousands of concurrent connections without ever waiting.

Blocking wastes thread time waiting. Non-blocking uses callbacks to continue work while I/O completes.
