Chapter 15

Testing Your Code

Catch bugs before they catch your users

~13 min read

Why Testing Matters

Code without tests is code that might break at any moment and you will not know until a user reports it. That is a terrible way to build software. Testing is how you sleep at night after a deploy. It is how you refactor code with confidence. It is how you stop being afraid of your own codebase.

Test-driven development (TDD) is the practice of writing a test before writing the actual code. The test fails initially (red), you write the minimum code to make it pass (green), then you clean up (refactor). The mantra is: Red, Green, Refactor. TDD forces you to think about what your code should do before you write it. The result is cleaner, more testable code and near-zero regression bugs.

The Three Levels of Tests

Unit Tests

A unit test tests a single function or method in isolation. You give it inputs, you assert the expected outputs. It does not talk to a database, call an API, or write to disk. It is pure logic. Unit tests are fast — thousands run in seconds. They form the foundation of the testing pyramid. You should have many of them.

Example: if you have a function formatTweet(text, maxLength) that truncates a tweet to fit, you write unit tests for every edge case. Normal text. Text at exactly the limit. Text one character over. Empty text. Null input. Each test takes milliseconds. If someone later breaks the function, you know instantly.

Integration Tests

Integration tests verify that multiple pieces work together. Does the API handler call the database correctly? Does the auth middleware pass the right user object to the controller? Do messages flow properly between services? These tests are fewer than unit tests because they are slower and more complex to set up. But they catch the bugs that unit tests miss — the ones that happen at the boundaries between components.

End-to-End Tests

E2E tests simulate a real user. They open a browser, click buttons, fill forms, and verify that the whole system behaves correctly. They are the slowest and most fragile tests. A network hiccup can make them fail. But they provide the highest confidence that the system actually works from the user's perspective.

At Twitter, an E2E test might log in as a test user, compose a tweet, post it, scroll the timeline, and verify the tweet appears. If that passes, the core loop works.

The Testing Pyramid

The pyramid shape tells you how many of each to write: lots of unit tests (the wide base), a moderate number of integration tests, and a few E2E tests (the narrow top). If you invert this — lots of E2E tests, few unit tests — your test suite will be slow, flaky, and a nightmare to maintain.

A healthy test suite gives you confidence. You push code, the tests run, they pass, you move on. You push code, a test fails, you know exactly what broke. No guessing. No manual checking. No users discovering bugs for you.

TDD in Practice

Here is how TDD works for a real feature. Say you need to write a function that calculates how many characters remain in a tweet. You start by writing a test: assert remainingChars("hello", 280) == 275. It fails because the function does not exist yet. Good. Now you write the function: def remainingChars(text, limit): return limit - len(text). The test passes. Now you refactor — maybe rename variables, extract logic — and the test still passes. You move to the next test case.

TDD forces incremental progress. Each test is a small step. You never write more code than needed to pass the current test. The result is high test coverage and minimal complexity.

Testing at Twitter

Before Twitter ships a new feature, hundreds of automated tests run. If even one test fails, the deploy stops. Here is how that saves them from disasters. Imagine a developer changes how tweets are stored. The change seems harmless, but it accidentally breaks the search feature. The unit test for tweet storage passes. But the integration test that searches for a tweet and verifies the result catches the break. The deploy halts. The developer fixes the bug before any user is impacted.

Without tests, that bug would reach production. Users would search for tweets and get nothing. They would report it. Someone would be paged at 2 AM to rollback the deploy. This is the difference between professional engineering and chaos. Tests are not optional. They are the safety net that lets you ship fast without fear.
