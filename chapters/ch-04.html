Chapter 04

How Data Flows Through a System

Controller, service, repository, database — why we split code into layers and what each layer actually owns

~11 min read

Why One Big File Isn't Enough

If you wrote an entire Twitter backend in a single file, it would work. The server would listen, route requests, query the database, and return responses. It would function perfectly for a while. But then you'd need to add a feature — maybe retweets. You'd scroll through thousands of lines to find where tweet creation happens, carefully edit it without breaking anything else, and pray you didn't introduce a bug. Then someone asks you to add logging of every tweet creation. Now you're hunting through the same file, trying to figure out where to insert the logging logic without duplicating it across seven different places.

This is why backend systems are organized into layers. Each layer has a specific job. The layers sit on top of each other like floors in a building. The top floor (controller) knows about the floor below it (service) but doesn't care about the foundation (database). This separation allows you to change one layer without touching the others — as long as the interface between them stays the same.

The four standard layers are: Controller (routes and HTTP concerns), Service (business logic and rules), Repository (data access and queries), and Database (persistent storage). Let's walk through each one with a concrete Twitter example: posting a tweet. We'll trace every step from the moment you tap "Post" to the moment your tweet is saved in the database.

The Controller Layer: The Traffic Cop

The controller (also called a route handler or endpoint) is the entry point for every request. Its job is limited to three things: receive the incoming request data, call the appropriate service, and send back the response. A controller should never contain business logic. It should never talk directly to the database. It is purely a coordinator.

Here's what the controller for creating a tweet might look like conceptually:

1. It receives the request object from the framework. The body contains JSON like {"text": "Hello world!"}.
2. It extracts the relevant data — the tweet text from the body, the authenticated user's ID from the request object (added by auth middleware).
3. It calls the TweetService with a simple command like tweetService.createTweet(userId, text).
4. It waits for the service to finish. The service returns the created tweet object.
5. It formats the response with status 201 Created and sends back the tweet as JSON.

That's it. The controller doesn't check whether the text contains banned words. It doesn't decide whether to store the tweet in a database or a cache. It doesn't know or care how the tweet is saved. It just says "Service, please handle this" and sends back whatever the service returns. If tomorrow Twitter changes its database from PostgreSQL to something else, the controller doesn't need to change at all.

In a real codebase, a controller is typically a short function — maybe 5 to 15 lines. If a controller is longer than that, it's probably doing too much. A good rule of thumb: if your controller needs a comment to explain what it's doing, that logic probably belongs in a service instead.

The Service Layer: The Brain of the Operation

The service layer (also called business logic layer) is where the actual rules of your application live. This is the most important layer to understand, because it's where your product's unique behavior is defined. The service layer is called by the controller, and it calls the repository layer. It sits in the middle and orchestrates everything.

When the TweetService receives the createTweet command from the controller, here's what it might do:

1. Validate business rules — Is the tweet text under 280 characters? (Twitter's limit). Does the user have posting privileges? (Not suspended, not rate-limited). Does the tweet contain spam or prohibited content? Each of these checks is a business rule that belongs in the service layer, not in the controller and definitely not in the database layer.

2. Enrich the data — The service adds metadata: a timestamp, a unique tweet ID (maybe a UUID), parsed URLs or mentions from the text, and potentially a reference to any media attached. This enrichment is business logic — it knows what a tweet is and how it should be structured.

3. Handle cross-cutting concerns — If posting a tweet should also update the user's tweet count, notify followers, and add to a search index, the service orchestrates all of this. It calls the user repository to increment the count, pushes a notification event, and sends the tweet data to the search index. The controller doesn't need to know about any of these side effects.

4. Call the repository — Once all business rules pass and the data is enriched, the service calls the TweetRepository: tweetRepository.save(tweetData). The service doesn't know or care whether the repository writes to PostgreSQL, MongoDB, or a plain text file. It just says "save this."

5. Return the result — The service returns the created tweet object (now with an ID and timestamp) back to the controller, which sends it to the client.

The service layer is where unit tests shine. Because the service doesn't depend on HTTP or the database directly (it depends on abstractions like the repository interface), you can test every business rule in isolation. You can verify that tweets over 280 characters are rejected, that suspended users can't post, that the correct notifications are triggered — all without running a server or a database.

The Repository Layer: The Data Gatekeeper

The repository layer (also called data access layer) is the bridge between your application code and your data storage. Its job is to provide a clean interface for reading and writing data while hiding the details of how that data is stored. The repository speaks the language of your application (objects, structs, hashes), not the language of the database (SQL queries, table rows).

When the TweetService calls tweetRepository.save(tweetData), the repository:

1. Takes the high-level tweet object and maps it to database columns: id, user_id, text, created_at, like_count.
2. Constructs a SQL INSERT statement (or a NoSQL document insert, depending on the database).
3. Executes the query using a database driver or ORM (Object-Relational Mapper).
4. Handles any database errors (like a unique constraint violation) and converts them into application-level errors.
5. Maps the database result back into an application object and returns it.

The repository layer is also where query optimization lives. If Twitter needs to speed up timeline loading, engineers optimize the queries in the repository layer — adding indexes, writing more efficient JOINs, or switching to a cache — without changing the service layer. The service still calls tweetRepository.findByUserId(userId), but behind the scenes, the repository might now check Redis first before hitting PostgreSQL.

A good repository pattern also makes it easy to swap databases. If Twitter migrates from PostgreSQL to a distributed database like Cassandra, they write a new repository implementation that follows the same interface. The service layer doesn't change at all. This is the power of abstraction: the service depends on an interface (a contract), not a concrete implementation.

The Database Layer: Where Data Sleeps

The database is the bottom layer — the actual storage system. This could be PostgreSQL (a relational database), Redis (an in-memory cache/key-value store), Cassandra (a wide-column NoSQL database), or even a simple file on disk. Twitter uses multiple databases for different purposes.

For the tweet example, the database does the actual persistence: it writes the tweet data to disk, ensures it's stored durably, maintains indexes so the tweet can be found quickly later, and enforces constraints (like ensuring the user_id references a real user). The database doesn't know about HTTP, doesn't know about business rules, doesn't know about the shape of a tweet as an object — it just knows about tables, rows, columns, and indexes.

An important concept here is a transaction. When the service layer needs to do multiple database operations that must all succeed or all fail together (like creating a tweet AND incrementing the user's tweet count), it wraps them in a database transaction. If either operation fails, the database rolls back both changes as if neither happened. This ensures data consistency even when things go wrong.

Why Layers Matter: The Twitter Example

Let's trace the full path of a tweet from tap to storage, now that you understand each layer:

1. Client: You tap "Post" on your iPhone. The app sends POST /api/v2/tweets with {"text": "Hello world!"} and your auth token in the header.

2. Controller: The tweet creation controller receives this. It extracts userId from the auth middleware and text from the body. It calls tweetService.createTweet(userId, text, mediaIds?).

3. Service: The service checks text length (max 280), checks user is not suspended, extracts @mentions and URLs, generates a unique ID. It calls tweetRepository.save(tweetData) and userRepository.incrementTweetCount(userId) inside a transaction. It also pushes a "new tweet" event to a notification queue so followers get notified.

4. Repository: The repository maps the tweet data to a SQL INSERT, executes it against PostgreSQL, catches any database errors, and returns the saved tweet object with its new ID and timestamp.

5. Database: PostgreSQL writes the row to disk, updates indexes so the tweet can be found in timeline queries, and returns the inserted row to the repository.

6. The response travels back up through the layers: repository → service → controller → client. Your phone displays the new tweet in your timeline. You see it immediately. The entire round trip took maybe 200 milliseconds, and every layer played its part without overstepping.

This layered architecture is what allows Twitter's engineering team — thousands of developers — to work on the same codebase without stepping on each other's toes. Frontend engineers work on the client layer. API engineers work on controllers. Product engineers work on services. Data engineers optimize repositories and databases. As long as the interfaces between layers remain stable, each team can change their layer independently.

And that's the real lesson of backend architecture: good code is code that can change without breaking everything else. Layers are the tool that makes this possible. They force you to think about responsibility, about separations, about contracts. They make your code maintainable, testable, and — most importantly — understandable to the next developer who has to read it.
