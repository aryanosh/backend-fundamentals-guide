How Data Flows Through a System

Controller, service, repository, database — why we split code into layers and what each layer owns

~11 min read

Why One Big File Isn't Enough

Layered architecture splits one messy file into four focused layers

When you're starting out, it's tempting to put everything in one file. A single script that handles HTTP requests, runs database queries, formats JSON responses, and sends emails. It works — for a while. Then the codebase grows, and that single file becomes a nightmare. A change in one place breaks something unrelated. You can't test individual pieces.

This is why production systems use layered architecture. Instead of one big blob, you split your application into distinct layers, each with a single responsibility. Each layer only talks to the layer directly below it. The most common pattern: Controller → Service → Repository → Database.

The Controller Layer: The Traffic Cop

The controller is thin — it validates, delegates, and responds. No business logic here.

The controller is the entry point for every request. It's the first layer that touches incoming data. Think of it as a traffic cop: its job isn't to know where every car is going, but to point each car in the right direction.

When your browser sends GET /api/search?q=backend, a controller catches that request. It parses the URL, extracts the query parameter, and checks if everything is well-formed. If something's wrong, it returns an error immediately. If everything looks good, it passes the sanitized data to the service layer.

The controller should be thin. It validates input, calls the appropriate service, and sends back a response. That's it. No database queries, no complex calculations.

The Service Layer: The Brain of the Operation

The service layer is where your application's intelligence lives

Below the controller sits the service layer. This is where your application's actual intelligence lives. If the controller is the traffic cop, the service is the strategist who decides where traffic should go and why.

The service contains business logic — the rules and decisions that define what your application does. A search service knows how to rank results by relevance. A payment service knows how to calculate tax and apply discounts. A notification service decides whether to send an email or a push notification based on user preferences.

Services orchestrate. They call multiple repositories, combine data from different sources, enforce business rules, and decide what the final result looks like. A well-written service is framework-agnostic — it doesn't know whether it's being called from an HTTP controller, a CLI tool, or a test suite.

The Repository Layer: The Data Gatekeeper

The repository translates clean method calls into database queries. Swap databases without touching services.

Once the service has decided what data it needs, it calls the repository layer. The repository is a clean abstraction over the database. It speaks the language of your application — methods like findUserById(id), saveOrder(order) — rather than raw SQL.

Why not just put database queries directly in the service? Because that couples your business logic to a specific database technology. If you switch from PostgreSQL to MongoDB, you'd only change the repository implementation — the service code stays untouched.

Repositories also provide a natural place for cross-cutting concerns like caching, retry logic, connection pooling, and query optimization. The service doesn't need to know the repository maintains an in-memory cache — it just calls findUserById and gets the result.

The Database Layer: Where Data Sleeps

Modern backends use multiple data stores behind a single repository interface

At the bottom of the stack is the database. This is where data actually lives — on disk, in memory, or spread across a cluster of machines. The database is the only layer that deals with raw persistence. Everything above it is just moving, shaping, and transforming data on its way in or out.

In modern systems, the "database" isn't always a single thing. You might have PostgreSQL for user accounts and orders, MongoDB for flexible content, Redis for fast caching, and Elasticsearch for full-text search. The repository layer hides this complexity — the service calls searchProducts(query) without knowing whether the repository hits a dedicated search index or falls back to a SQL query.

Why Layers Matter: The Google Example

Google's search architecture follows the same layered pattern at massive scale

Think about what happens when you search on Google. The controller receives your GET /search?q=latest+news request. The search service orchestrates a complex pipeline: it calls the QueryRepository to check cached results, the RankingService to score results, the PersonalizationService to factor in your history, and the AdService to pick relevant ads.

Each of these is a separate layer or service with a clear boundary. Google's engineers can optimize the ranking algorithm without touching the ad system. They can swap out the caching layer from Memcached to Redis without changing any business logic. They can add a new frontend (like voice search) by writing a new controller that calls the same services.

That's the power of layered architecture. It's not extra work — it's insurance against the chaos that every growing codebase inevitably faces.
