How Data Flows Through a System

Controller, service, repository, database — why we split code into layers and what each layer owns

~11 min read

Why One Big File Isn't Enough

Production systems use layered architecture to keep codebases maintainable. Instead of one file handling everything, each layer has a single responsibility.

  • One big file — Mixes HTTP handling, business logic, DB queries, auth, emails in one place
  • Layered architecture — Splits into Controller, Service, Repository, Database
  • Each layer — Only talks to the layer directly below it
Layered architecture splits one messy file into four focused layers

The Controller Layer: The Traffic Cop

The controller is the entry point for every request. It validates input, calls the appropriate service, and sends back a response. No business logic, no database queries.

  • Parses — Extracts method, path, headers, body from the request
  • Validates — Checks input is well-formed, returns 400 if not
  • Delegates — Calls the service layer, then formats the response
The controller is thin — it validates, delegates, and responds. No business logic here.

The Service Layer: The Brain of the Operation

The service layer contains your application's business logic. It orchestrates repositories, enforces rules, and makes decisions. A well-written service is framework-agnostic.

  • Business rules — Defines what your application does and why
  • Orchestration — Calls multiple repositories, combines data from different sources
  • Framework-agnostic — Works the same from HTTP, CLI, or tests
The service layer is where your application's intelligence lives

The Repository Layer: The Data Gatekeeper

The repository abstracts the database behind clean method names. It speaks your application's language — not raw SQL.

  • Abstraction — Methods like findUserById(id), saveOrder(order) hide SQL
  • Swapability — Switch databases by changing only the repository implementation
  • Cross-cutting — Caching, retry logic, connection pooling live here
The repository translates clean method calls into database queries.

The Database Layer: Where Data Sleeps

The database is where data actually lives — on disk, in memory, or across a cluster. Modern systems use multiple data stores behind a single repository interface.

  • Persistence — The only layer that deals with raw storage
  • Multiple stores — PostgreSQL for accounts, Redis for cache, MongoDB for content
  • Hidden complexity — Service calls searchProducts(query) without knowing where data lives
Modern backends use multiple data stores behind a single repository interface

Why Layers Matter: The Google Example

Google's search pipeline follows the same layered pattern. The controller receives your query, the search service orchestrates ranking, personalization, and ads — each in its own layer.

  • Independent optimization — Engineers improve ranking without touching ads
  • Swappable components — Swap caching layer without changing business logic
  • Multiple frontends — New interfaces reuse existing services
Google's search architecture follows the same layered pattern at massive scale
