Chapter 04

How Data Flows Through a System

Controller, service, repository, database — why we split code into layers and what each layer actually owns

~11 min read

Why One Big File Isn't Enough

When you're just starting out, it's tempting to put everything in one file. A single Python script that handles HTTP requests, runs database queries, formats JSON responses, and sends emails. It works. For a while. Then the codebase grows, and that single file becomes a nightmare. A small change in one place breaks something seemingly unrelated. You can't test individual pieces. You can't reason about the code without holding the entire thing in your head at once.

This is why production backend systems use layered architecture. Instead of one big blob of code, you split your application into distinct layers, each with a single responsibility. Each layer only talks to the layer directly below it. This creates a clean, one-way flow of data that's easy to understand, test, and modify.

The most common pattern you'll encounter — powering everything from Google's backend to a startup's Node.js API — is a four-layer architecture: Controller → Service → Repository → Database. Each layer is a walled garden that does one job and does it well.

The Controller Layer: The Traffic Cop

The controller is the entry point for every request. It's the first layer that touches incoming data. Think of it as a traffic cop standing at the intersection: its job isn't to know where every car is going, but to point each car in the right direction.

When your browser sends GET /api/search?q=backend to Google, a controller catches that request. It parses the URL, extracts the query parameter q, and checks if the request is well-formed. Did you include the required headers? Is the authentication token valid? If something's wrong, the controller returns an error immediately — before any business logic runs. If everything looks good, it passes the sanitized data to the service layer.

The controller should be thin. It validates input, calls the appropriate service, and sends back a response. That's it. No database queries, no complex calculations, no email sending. If a controller is more than a few dozen lines, something is wrong.

The Service Layer: The Brain of the Operation

Below the controller sits the service layer. This is where your application's actual intelligence lives. If the controller is the traffic cop, the service is the strategist who decides where traffic should go and why.

The service contains business logic — the rules and decisions that define what your application does. A search service knows how to rank results by relevance. A payment service knows how to calculate tax and apply discounts. A notification service knows whether to send an email or a push notification based on user preferences.

Services orchestrate. They call multiple repositories, combine data from different sources, enforce business rules, and decide what the final result should look like. A well-written service is framework-agnostic — it doesn't know or care whether it's being called from an HTTP controller, a command-line tool, or a test suite. This makes services reusable and testable in isolation.

The Repository Layer: The Data Gatekeeper

Once the service has decided what data it needs, it calls the repository layer. The repository is a clean abstraction over the database. It speaks the language of your application — methods like findUserById(id), saveOrder(order), deleteExpiredSessions() — rather than raw SQL or query builder syntax.

Why not just put database queries directly in the service? Because that couples your business logic to a specific database technology. If you ever switch from PostgreSQL to MongoDB, or from MySQL to DynamoDB, you'd have to rewrite every service that touches the database. With a repository layer, you only change the repository implementation — the service code stays untouched.

Repositories also provide a natural place for cross-cutting data concerns like caching, retry logic, connection pooling, and query optimization. The service doesn't need to know that the repository maintains an in-memory cache of frequently accessed users — it just calls findUserById and gets the result.

The Database Layer: Where Data Sleeps

At the bottom of the stack is the database. This is where data actually lives — on disk, in memory, or spread across a cluster of machines. The database is the only layer that deals with raw persistence. Everything above it is just moving, shaping, and transforming data on its way in or out.

In modern systems, the "database" isn't always a single thing. You might have a primary relational database (PostgreSQL) for user accounts and orders, a document store (MongoDB) for flexible content, a key-value cache (Redis) for fast lookups, and a search index (Elasticsearch) for full-text queries. The repository layer hides this complexity — the service calls searchProducts(query) without knowing whether the repository hits a dedicated search index or falls back to a SQL LIKE query.

Why Layers Matter: The Google Example

Think about what happens when you search on Google. The controller receives your GET /search?q=latest+news request. The search service orchestrates a complex pipeline: it calls the QueryRepository to check for cached results, the RankingService (another service!) to score results by relevance, the PersonalizationService to factor in your search history, and the AdService to pick relevant ads.

Each of these is a separate layer or service with a clear boundary. Google's engineers can optimize the ranking algorithm without touching the ad system. They can swap out the caching layer from Memcached to Redis without changing any business logic. They can add a new frontend (like voice search) by writing a new controller that calls the same services.

That's the power of layered architecture. It's not extra work — it's insurance against the chaos that every growing codebase inevitably faces.
